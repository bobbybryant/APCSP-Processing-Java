01 · why loops exist

The "don't repeat yourself" machine

Imagine you need to print the numbers 0 through 4. You could write five println() lines. But what about 0 to 1000? A loop says the boring part once and tells the computer how many times to repeat it.

Two flavors, one idea

    A for loop is best when you know how many times to repeat. A while loop is best when you repeat until something becomes true. They can both do the same jobs — but each is cleaner for one situation. We'll meet both.
  







  02 · anatomy
  
The for loop, part by part

  
A for loop packs three jobs into one line, separated by semicolons. Read it as: "start here; keep going while this is true; do this between each run."


  


  

    
PartExampleWhen it runsIts job

    
start / initint i = 0once, at the very beginningcreates the counter and sets its first value

    
condition / testi < 5before every passif true → run the body. if false → stop.

    
updatei = i + 1after every passchanges the counter so the loop eventually ends

    
body{ println(i); }each pass the test is truethe actual work being repeated

  


  

    
The order is fixed

    init → test → body → update → test → body → update → … The test is always checked first, before the body. That's why a loop can run zero times if the test is false from the start.
  


  
Reading it out loud

  
Get in the habit of narrating: "i starts at 0. Is 0 less than 5? Yes — print it. Add 1. Is 1 less than 5? Yes…" When you can say it, you can count it — which is exactly what tomorrow's quiz asks.

  04 · the tricky details
  
< vs <= and how big the step is

  
Two tiny choices change the answer completely: whether the limit is included, and how much the counter jumps each pass. These are where off-by-one mistakes live.


  
Does the limit count? (< vs <=)

  

    

      

    

    

      

    

  

  
<= lets the limit through the door. < slams it shut one early.


  
Steps bigger than 1

  
The update doesn't have to be + 1. Counting by 2s, 3s, or backwards is common — and it changes the count. To count the runs, list the actual values the counter takes:

  


  

    
Counting-by-N shortcut

    For a clean range, runs ≈ (end − start) ÷ step, then check the endpoint by hand. But when steps don't divide evenly, always list the values — the shortcut can mislead you by one.
  


  
Counting down

  

  
Counting down only works if the update decreases the counter (a negative step) and the test points the right way (>= or >). Mix those up and you get the trap in the next section.







  05 · the trap
  
When a loop never stops

  
A loop ends only when its test eventually becomes false. If the update never moves the counter toward failing the test, the loop runs forever — your sketch freezes. Three classic causes:


  
1 · The update moves the wrong direction

  

  
The test wants i to eventually reach 10, but the update keeps subtracting. The counter races away from the finish line. Drop the "careful…" preset into the trace visualizer above to see it flagged.


  
2 · The counter never changes (missing update)

  


  
3 · The condition can never be false

  


  

    
How to spot it fast

    Ask one question: "does the update push the counter toward failing the test?" Counting up should pair with < / <=; counting down should pair with > / >=. If the arrow and the test disagree — or the counter never moves — it's infinite.
  







  06 · condition-driven
  
The while loop

  
A while loop is the bare-bones loop: just a condition and a body. It keeps running as long as the condition is true. You manage the counter yourself — before the loop, and inside it.


  


  

    
The for loop's three parts, scattered

    A while loop has the exact same three jobs as a for loop — start, test, update — but they're not lined up on one line. The start sits above the loop and the update sits inside the body. Forget the update and you've built an infinite loop.
  


  
It can run zero times… or exactly once

  
Because the test is checked first, a while loop can run zero times if the condition starts false. And a condition that's only briefly true runs a tiny number of times:

  


  
Predict the output

  

  
Trace it the same way you trace a for loop: test, body, update, repeat. The <= is the detail to watch — it lets 5 print before the test finally fails at 6.







  07 · which one?
  
Choosing for vs while

  
They can both do the same jobs. The question is which one makes your intent obvious to the next person reading the code (often future you).


  

    
 for loopwhile loop

    
best forrepeating a known number of times ("do this 10 times")repeating until a condition is met ("until the user clicks")

    
counterbuilt into the loop headeryou manage it yourself, by hand

    
reads like"for each value from A to B…""while this is still true…"

    
typical usedrawing 12 evenly-spaced shapeslooping while a value stays in range

  


  

    

      
for = counting

      If you can say the number out loud before the loop starts ("five times", "once per column"), reach for for. It keeps the counter logic in one tidy place so you can't forget the update.
    

    

      
while = condition

      If you're waiting for something to happen and don't know the count in advance, reach for while. Just remember you own the counter — set it before, change it inside.
    

  


  
Same destination, different signpost. Pick the one that tells the truth about why you're looping.







  08 · the remainder operator
  
Modulo: % gives the leftover

  
The % operator divides and hands you the remainder — what's left over after the division. 17 % 5 is 2, because 5 goes into 17 three times with 2 left over. It looks strange at first, but it unlocks two everyday tricks.


  


  
Trick 1 · even or odd

  
A number is even exactly when dividing by 2 leaves nothing. So n % 2 is your even/odd test — and notice it uses == (compare), never = (assign):

  

  

    
Two classic mistakes

    n % 2 = 1 tries to assign and won't compile — you need ==. And don't flip the meaning: remainder 0 is even, remainder 1 is odd. Mixing those up is a favorite trick question.
  


  
Trick 2 · "every Nth time"

  
Inside a loop, i % 3 == 0 is true only when i is a multiple of 3 — so you can do something special every 3rd pass. Great for stripes, grids, and patterns.

  09 · controlled chaos
  
random() and casting with int()

  
random() hands back a float — a decimal. To get a whole number you can count on (a shape size, a grid position), you wrap it in int() to chop off the decimal. This "casting" is the part people forget.


  


  

    
Why 21 and not 20?

    random(low, high) can return values up to — but never reaching — high. After int() chops the decimal, the biggest whole number you can get is one below the top. So int(random(10, 21)) gives you 10 through 20 inclusive. Want 1–6 like a die? Use int(random(1, 7)).

  11 · debugging drill
  
Bug hunt: fix the broken loop

  
This loop is supposed to draw a row of 5 circles marching across the screen — but it's riddled with errors. Find them. Click each clue to check your thinking. This is exactly the debugging muscle the quiz tests.


  


  
BUG 1 · the counter has no type
    
x appears in the loop but was never declared.

    
    
A loop counter has to be created with a type the first time it's used. The start part should be int x = 0, not just x = 0. Without int, the program doesn't know what x is and won't compile.

  


  
BUG 2 · the test is an assignment
    
x = 5 in the middle slot uses a single equals sign.

    
    
The middle slot of a for loop needs a true/false test, but x = 5 tries to assign 5 to x. To draw 5 circles (positions 0–4) you want x < 5. Remember: = assigns, == and < compare.

  


  
BUG 3 · the update runs the wrong way
    
x = x - 1 makes the counter shrink.

    
    
Starting at 0 and subtracting, x goes 0, −1, −2… and never reaches 5, so the loop never ends (and the circles march off the left edge). To count up toward the limit, the update must add: x = x + 1.

  


  
BUG 4 · a missing semicolon
    
The ellipse(...) line has no semicolon.

    
    
Every statement ends with a semicolon. The ellipse(50 + x * 40, 100, 30, 30) call needs a ; at the end or the sketch won't compile.

  


  
★ the corrected version
    
    

      

      Declared counter (int x), a real test (x < 5), an update that counts up (x = x + 1), and a semicolon on the statement. The result: 5 circles at x = 50, 90, 130, 170, 210.
    

  







  12 · check yourself
  
Practice it (same formats as the quiz)

  
Different numbers, same skills. Click an answer to check it, then read why. No peeking until you've traced it on paper.


  
Q1 · how many times?
    

    

    
x takes 0, 2, 4 — then 6 is not < 6, so it stops. 3 times. Listing the values beats guessing.

  


  
Q2 · how many times?
    

    

    
i takes 3, 6, 9 — and <= lets 9 through. 3 times.

  


  
Q3 · how many times?
    

    

    
i starts at 5 and keeps growing (6, 7, 8…), so i > 0 is always true. The update pushes away from failing the test → infinite.

  


  
Q4 · how many times?
    

    

    
Is 2 <= 2? Yes → run once, print 2, i becomes 3. Is 3 <= 2? No → stop. 1 time.

  


  
Q5 · what is the output?
    

    

    
Starts at 1, the <= includes 4, stops when x reaches 5. Output: 1 2 3 4.

  


  
Q6 · never possible?
    

    
Which can never be printed?

    

    
"Apple" needs x < 0 AND x > 50 at the same time — impossible. So Apple can never appear. Trace each path: a value can't be both negative and above 50.

  


  
Q7 · modulo
    
What does 15 % 2 evaluate to, and is 15 even or odd?

    

    
2 goes into 15 seven times (14) with 1 left over. Remainder 1 → odd.

  


  
Q8 · write a FOR loop
    
Print 0, 1, 2, 3 — each on its own line. Try it on paper first, then check one valid answer.

    
    

      

      Start at 0, stop before 4 (so 0–3), step by 1. println already starts a new line each call. Using i <= 3 works too.
    

  


  
Q9 · write a WHILE loop
    
Count from 100 to 115 by 5s, including 115, one per line. Try it, then check.

    
    

      

      Set the counter before the loop, use <= so 115 prints, step by 5, and update inside the body. Values: 100, 105, 110, 115.
