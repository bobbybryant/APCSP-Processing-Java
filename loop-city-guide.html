00 · the mission

What the City Council is actually asking for

Strip away the story and the job is this: build a 1400 × 800 scene out of reusable parts. The assignment already hands you three function signatures — that's a gift, not a suggestion. Your structures must be drawable anywhere on the block just by changing the numbers you pass in.

The three you're given:

drawHospital(int x, int y)   triangular roof + a red cross

drawOffice(int x, int y, int numFloors)   height grows with the floor count

drawThird(int x, int y)   a structure of your choosing

Then one main loop places them along a road, choosing the type with a modulo pattern. Offices roll random(3, 7) floors.

Things we've already covered that you'll lean on
Functions with parameters · the for loop · the % modulo operator · if/else if chains · random() · drawing order with rect, triangle, circle, fill.
01 · decomposition

Think in building blocks, not in one giant pile

The whole point of the assignment is in its name. A good Loop City has a tiny setup() that reads almost like a to-do list, and the messy details live inside helper functions. If setup() is 200 lines long, something's gone sideways.

Ask yourself
  • What is the one job of setup() here — to draw rectangles, or to arrange things that already know how to draw themselves?
  • If I wanted to drop a hospital at a totally different spot tomorrow, what's the only thing that should need to change?
  • Inside drawOffice, every rect and circle is positioned relative to what? (Hint: not the absolute screen — the x and y that got passed in.)

That last point is the secret to a reusable function: everything inside is measured from the parameters, never from a fixed screen coordinate. A window isn't at "x = 320." It's at "the building's x, plus 80."

Watch out
If you find yourself typing a specific number like 600 inside a draw function, pause. Should that be x + 600 instead? A function that hard-codes screen positions can only ever draw one building, in one place.
02 · layering

Whatever you draw last lands on top

Processing paints like a stack of stickers — each shape covers whatever's underneath it. Get the order wrong and your buildings vanish behind the grass, or the sun hides behind a cloud that should be in front. Step through the layers below and watch the scene assemble.

Painter's order — press Paint Next Layer
Ask yourself
  • If the grey ground is drawn after the buildings, what happens to the buildings?
  • Should the road's dashed lines come before or after the grey road surface? Why?
  • Where in the order does the sky belong — and what would it cover if you drew it late?
03 · the engine

One loop, three buildings: the modulo dispatcher

This is the heart of the assignment. A single for loop counts 0, 1, 2, 3, … and on each pass you decide which building to draw. The trick is the remainder operator %: dividing the counter by 3 gives you a remainder that cycles 0, 1, 2, 0, 1, 2, … forever. That repeating cycle is your pattern.

Modulo dispatcher — drag the counter
loop counter  x = 0
Why remainders cycle
0%3=0, 1%3=1, 2%3=2, then 3%3=0 again. The remainder can only ever be 0, 1, or 2 — exactly three buckets, one per building type. That's why % 3 and not % 2 or % 4.
Ask yourself
  • You have three remainder cases. What structure have we used that runs one branch out of several based on a condition? (Think if … else if … else if.)
  • If you wanted the pattern hospital → office → store, which remainder should trigger which function?
  • What would % 4 do to your pattern — and would you have a fourth building to fill it?
04 · spacing

Same width makes the math almost disappear

Here's the rule I keep hammering: your loop starts at 0 and changes by 1. No clever x = x + 250 in the loop header. The counter just counts; you do the spacing math inside, every single time, from the counter. And it's far easier if every building is the same width — pick one width (250, 300, whatever) and stick to it.

Then the left edge of building number i is simply width × i (plus maybe a small starting offset). Slide the controls and watch the positions fall out of the counter.

Position calculator — width × counter
building width 220
how many 5
Watch out
If you let widths differ, width × i stops working and you'll need a running "where does the next one start" variable instead. Doable — but you've just made the assignment harder. Keep them equal and the multiply does all the work.
Ask yourself
  • If width is 220 and the counter is 3, what's the left edge? Did you get 660? Now where's building 0?
  • Why must the spacing math happen inside the loop body and not in the loop header?
05 · the nested loop

Stacking floors — and the off-by-one that trips everyone

The office is the only building whose height changes, so it needs its own loop, living inside drawOffice — a loop inside a function. The ground floor is special (it gets a door and a bush), and every floor above it is a copy of the same window-floor, stacked higher.

Read the spec carefully: "if floors is 3, there's a ground floor and then 2 floors of windows." So the number of window floors is numFloors − 1, not numFloors. Slide it and watch the count.

Floor stepper — what does numFloors actually build?
numFloors = 3
Ask yourself
  • Your floor loop counts from 0 and changes by 1 (always). How do you turn "floor number" into a height offset so each floor sits on top of the last? (Multiply inside, like before.)
  • Should the ground floor be drawn by the loop, or before it? Why might keeping it separate be cleaner?
  • The random trap: the spec wants 3 to 7 floors. What numbers can int(random(3, 7)) actually return? Try it on paper — does 7 ever appear? If you need 7 included, what should the second number be?
Loop-inside-a-function is still just a loop
Same rules: start at 0, step by 1, do the math in the body. The only new idea is that this loop's count comes from a parameter (numFloors) instead of a fixed number.
06 · the reference line

Anchor everything to the road with negative heights

Buildings of different heights all need to sit on the same ground line. The clean trick — and the one I recommended — is to treat the road's top edge as your baseline y, and grow shapes upward using a negative height. Then no matter how tall a building is, its feet stay glued to the road.

Ask yourself
  • When the height grows upward from the baseline, which coordinate stays fixed — the top of the building or the bottom?
  • If you grew buildings downward from a fixed top instead, what would happen to a short building versus a tall one along the road?
  • Inside the office, every floor stacks above the one below. How does a shared baseline make that stacking predictable?
07 · finishing touches

The dashed road is just one more loop

The center line of a road is a row of evenly spaced dashes — which is exactly what a loop is good at. Same rules apply: count from 0, step by 1, compute each dash's position from the counter.

Ask yourself
  • How wide is one dash, and how big is the gap after it? Together those decide the spacing per step.
  • If each dash is at spacing × i, how many dashes do you need to cross a 1400-wide screen?
  • The road itself is a single rectangle. Should it be drawn before or after the dashes sit on top of it?
08 · before you submit

Planning checklist

Tick these off as you build. If you can't tick one yet, you've found your next task. (This is just for you — nothing is saved.)

  • ✓size(1400, 800) and a background are set in setup().
  • ✓Three separate draw functions exist, each taking an x and y.
  • ✓Every shape inside a draw function is measured from its parameters, not fixed screen numbers.
  • ✓The hospital has a triangular roof and a red cross.
  • ✓The office uses its own loop to stack numFloors − 1 window floors above a ground floor.
  • ✓Offices pick a random floor count in the 3–7 range.
  • ✓One main loop places buildings, counting from 0 and stepping by 1.
  • ✓A % 3 modulo pattern decides which building each pass draws.
  • ✓The road has dashed lines made with a loop.
  • ✓Drawing order is correct — sky and ground are behind the buildings.
09 · if you're flying

Stretch goals to give the city life

Finished early? These are optional and they nudge you toward ideas we'll meet later — no pressure, just fun.

A car that drives down the street
  • What single variable would have to change a little every frame to make a car appear to move?
  • When the car runs off the right edge, how would you send it back to the start? (What does it mean for its x to "wrap around"?)
  • Harder: to drive one way and then back, what would have to flip — the position, or the amount you add each frame?
Clouds and other signs of life
  • You already have a function that draws a cluster of circles. Could it become a reusable drawCloud(int x, int y) you call a few times?
  • Where in the draw order do clouds belong so they float in front of the sky but behind nothing important?

11 · the recipe card

Questions to ask yourself, start to finish

No code here on purpose. If you can answer every line below in your own words, you can write Loop City.

Loop City — the planning recipe ✎
  • Decompose → What's the one job of setup(), and what belongs in helper functions instead?
  • Reuse → Is every shape inside a function measured from its x, y parameters?
  • Order → Am I drawing back-to-front — sky, ground, road, then buildings on top?
  • Dispatch → Does my main loop count 0, 1, 2, … and use % 3 to choose the building?
  • Branch → Three remainders, three cases — which control structure runs one branch of several?
  • Spacing → Same width for all? Then is each left edge just width × counter, computed inside the loop?
  • Floors → Does the office loop stack numFloors − 1 window floors above the ground floor?
  • Random → What values can int(random(3, 7)) really produce — is 7 included?
  • Anchor → Do my heights grow upward from the road baseline so feet stay on the road?
  • Loop rule → Every loop: start at 0, change by 1, do the math in the body.
Plan it on paper first. The code is just you typing up a plan you already made. ✦
