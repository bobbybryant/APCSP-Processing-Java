01Two different things

The Internet vs. the World Wide Web

People use these words interchangeably. On the AP exam (and in real life), they are not the same thing — and the difference is one of the easiest free points you'll ever earn.

The Internet = the roads

A network of networks

Physical hardware plus the shared rules for using it: fiber-optic cables under the ocean, routers, cell towers, Wi-Fi, satellites, and the protocols that let billions of separate networks act like one giant one. It moves any kind of data — web pages, but also video calls, game data, email, app traffic.

The Web = the deliveries

Information that travels on it

The World Wide Web is one service that runs on top of the Internet: pages, text, images, and links, requested with HTTP and viewed in a browser. The Web is content and links; the Internet is the system that carries them. No Internet → no Web. But the Internet would exist just fine without the Web.

exam-speak, one sentence each

Internet: a global, interconnected network of networks built on open protocols.
World Wide Web: a system of linked pages, programs, and files (information) accessed over the Internet using HTTP.

02Addresses & rules

IP Addresses & the Internet Protocol

For a letter to arrive, the envelope needs an address. For data to arrive, every device on the Internet needs one too: an IP address (Internet Protocol address). When your Chromebook joins the school Wi-Fi, it gets one. The server hosting this page has one. Routers use these addresses to decide where to send data next.

Build an IPv4 address

192.168.4.27
byte 1 192 11000000
byte 2 168 10101000
byte 3 4 00000100
byte 4 27 00011011
4 bytes × 8 bits = 32 bits total
wait, you've seen this before

Each chunk of an IPv4 address runs from 0 to 255 — that's exactly one byte, the smallest data type from our Data Types unit. An IPv4 address is just four bytes glued together with dots. Four bytes = 32 bits = about 4.3 billion possible addresses, and we've basically run out — which is why the newer IPv6 uses 128 bits (enough to give an address to every grain of sand on Earth, several times over).

So what's a protocol?

A protocol is a set of rules that lets devices made by different companies, in different countries, running different software, still understand each other. Two matter most for us:

IP — Internet Protocol

The addressing and routing rules. Every chunk of data gets a header listing the sender's and receiver's IP addresses — like the to/from on an envelope. Routers read only the header and pass the data toward its destination. IP does its best, but it makes no promises that anything arrives.

TCP — Transmission Control Protocol

The reliability rules, working on top of IP. TCP numbers every piece, confirms each arrival, re-requests anything missing, and reassembles everything in order at the far end. Together they're called TCP/IP — the rulebook nearly all Internet traffic follows.

why "open" matters

TCP/IP is an open standard: published for free, owned by no company. Anyone can build a device that speaks it — which is exactly why the Internet could grow from a handful of university computers to billions of devices without anyone's permission. Keep that thought for Section 04.

TCP vs. UDP — two ways to ship packets

IP gets packets addressed, but it makes no promise they arrive. The job of caring (or not caring) falls to one of two transport protocols riding on top of IP. Same Internet, two very different promises:

TCP — careful & complete

Numbers every packet, waits for a confirmation, and re-sends anything that goes missing, then reassembles everything in order. Slower, but nothing is lost. Used when every byte must be perfect: web pages, email, file downloads, texting.

UDP — fast & “good enough”

Fires packets as fast as possible and never looks back — no confirmations, no re-sends. A few dropped packets are fine. Used when being on time beats being complete: live video calls, streaming, online games.

the one-line difference

On a laggy video call the picture glitches for a second and then keeps going — that’s UDP choosing to skip the late packets rather than freeze. If that were a file download (TCP), it would instead pause and wait, because a file with missing bytes is broken.

From names to numbers: DNS

You never type 142.250.72.14 into your browser — you type a name. But routers only understand numbers. The Domain Name System (DNS) is the Internet’s phone book: it translates a human-friendly domain name into the IP address the network actually routes to.

03Break it, watch it heal

Packets, Routing & Fault Tolerance

Your message doesn't travel as one solid lump. It gets chopped into packets — small, numbered pieces, each carrying the destination address — and each packet finds its own path through the network. Different packets from the same message can take totally different routes and arrive out of order; TCP puts them back together at the end.

Why bother? Because the network has redundancy: many extra paths between any two points. If a router dies mid-delivery, packets simply route around it. A system that keeps working when parts fail is called fault-tolerant.

the punchline

Notice there's no single router the message depends on — and no headquarters deciding routes. Each router independently forwards packets toward their destination using whatever paths still exist. That decentralized, redundant design is why the Internet is fault-tolerant. (It was literally designed in the Cold War era to survive losing chunks of itself.)

vocab lock-in

Packet: a small chunk of data + metadata (addresses, order number). Routing: finding a path for each packet, hop by hop. Redundancy: extra connections beyond the minimum needed. Fault tolerance: the system still works when components fail — because of redundancy.

what a packet reveals even when it’s locked

Every packet still needs a readable header — source address, destination address, size, timestamp — so routers can deliver it. That outside-the-envelope information is called metadata. Even if the contents are encrypted (HTTPS, Section 09), anyone watching the wire can still log which sites a device talks to and when. That’s exactly how a school or an ISP can monitor browsing without ever reading the messages themselves.

04Built to grow

Why the Internet Scales

Scalability means a system can grow — more users, more devices, more traffic — without being redesigned. The Internet went from 4 connected computers in 1969 to tens of billions of devices today running on the same basic design. Three reasons why:

Open protocols

TCP/IP is free and public. A new device joins by following the rules — no permission, no fee, no central sign-up. Growth is bottom-up.

No central computer

There's no "main server of the Internet" to overload. Routing decisions happen locally at millions of routers, so adding devices doesn't pile work onto one machine.

Hierarchy

Addresses and networks nest inside bigger networks (your laptop → home router → ISP → backbone). Routers only need to know the next hop, not a map of everything.

05Three numbers, three meanings

Bandwidth vs. Bit Rate vs. Latency

"My internet is slow" actually mixes up three different measurements. Untangling them tells you what kind of slow you have — and what would actually fix it.

TermWhat it measuresHighway versionUnits
Bandwidth The maximum capacity of a connection — the most bits that could move per second. Number of lanes on the highway. Mbps / Gbps
Bit rate How many bits actually move per second right now. Always ≤ bandwidth (congestion, weak Wi-Fi, busy servers). How fast traffic is actually flowing. bps / Mbps
Latency The delay before data starts arriving — travel time for one round trip. How long the on-ramp light makes you wait. ms

The race: fiber vs. satellite

Two connections: Fiber (500 Mbps bandwidth, 12 ms latency) and GEO Satellite (150 Mbps bandwidth, 620 ms latency — the signal travels ~35,000 km into space and back). Pick a task and see which number decides the winner.

what decided it

the rule of thumb

Small or back-and-forth data → latency wins. Gaming, video calls, and clicking links feel snappy or laggy based on milliseconds of delay. Big one-way data → bandwidth wins. Downloads and streaming care about capacity. A 500 Mbps plan with terrible latency still feels awful for gaming — and that's why "more bandwidth" doesn’t fix every problem.

What actually carries the bits?

Bandwidth and latency describe how much and how fast — but the physical medium carrying the signal sets the ceiling. Four common ones, fastest-and-furthest first:

MediumHow it carries dataTrade-off
Fiber-opticPulses of light through glass strands.Highest bandwidth, lowest loss over long distance — the backbone of the Internet. Costly to install.
Copper wireElectrical signals (older phone/cable lines).Cheap and everywhere, but the signal weakens faster over distance and carries less.
Radio / wirelessRadio waves through the air (Wi-Fi, cellular, 5G).No cables — convenient and mobile, but shared, and walls and distance cause interference.
SatelliteRadio waves bounced off a satellite in orbit.Reaches places no cable can, but the trip up to orbit and back adds big latency.
don’t mix up “fast medium” and “low latency”

For sending data over a long distance, fiber-optic is the fastest medium — light through glass loses almost nothing on the way. Satellite’s problem isn’t a slow medium (its signal is also radio at light-speed); it’s the sheer distance up to orbit and back. That’s a latency issue, not a bandwidth one.

06Who gets left out

The Digital Divide

Everything on this page assumes you have the Internet. Billions of people don't — or have access so limited it barely counts. The digital divide is the gap between people who have full access to computing devices and the Internet, and people who don't.

The divide isn't one single gap. It runs along several lines at once:

Geography

Cities get fiber; rural areas may only have slow DSL, expensive satellite, or nothing. Running cable to remote places costs more than companies earn back.

Income

Monthly plans, devices, and data caps cost real money. A family may share one phone — and a phone plan is not the same as a laptop with home broadband.

Infrastructure & policy

Some regions lack reliable electricity, cell coverage, or competition between providers. Government choices about funding and regulation shape who gets connected.

why this is a CS topic, not just a social-studies topic

School, jobs, banking, healthcare, and government services keep moving online — so unequal access turns into unequal opportunity. AP CSP expects you to discuss how the digital divide affects who benefits from computing innovations, and how design choices (low-bandwidth modes, offline features, mobile-first pages) can widen or shrink the gap.

Debate-starters

There's no single right answer — that's the debate. Consider: can you fully participate in school, work, or government services today without it? Utilities are usually regulated so everyone can get them at fair prices. What would change if the Internet were treated that way?
Arguments both ways: the school upgrade helps every student equally — but only during school hours. Hotspots help fewer people but reach the exact place the divide lives: home, where homework happens. Strong answers weigh who is being left out and when.
It helps, but only partly. Try writing an essay, coding a Processing sketch, or filling out a job application on a 6-inch screen with a data cap. Quality of access — device, speed, data limits, reliability — matters as much as access itself.
07Smaller files, two philosophies

Compression: Lossless vs. Lossy

Every photo, song, and page you send gets compressed — rewritten to use fewer bits — because fewer bits means faster transfer and less storage. There are exactly two philosophies, and the difference is one question: can you get the original back, bit for bit?

Lossless — yes, perfectly

Finds patterns and repetition and describes them more efficiently. Decompressing rebuilds the exact original — every bit identical. Used when any change would be a disaster: text, code, spreadsheets, ZIP files, PNG images.

Lossy — no, and that's the point

Throws away details humans barely notice — tiny color differences, sounds masked by louder sounds. Much smaller files, but the discarded data is gone forever. Used for media where "looks/sounds the same" is good enough: JPEG, MP3, streaming video.

08Locking without meeting

Public & Private Key Encryption

Here's the problem: packets cross dozens of routers owned by strangers, and anyone along the path can read them — like sending a postcard. Encryption scrambles the message so only the intended reader can unscramble it. But that creates a puzzle: if Alice and Bob have never met, how do they agree on a secret key without an eavesdropper hearing it?

Two families of encryption

Before the clever part, the basic split. All encryption falls into one of two families — and the difference is simply how many keys are involved:

Symmetric — one shared secret key

The same single key locks and unlocks. Fast and simple — like a house key both people own a copy of. The catch: that key must stay secret, so how do you get a copy to the other person across an Internet full of eavesdroppers without it being stolen on the way?

Asymmetric — a public + private pair

Two different mathematically linked keys: one to lock, a different one to unlock. Nothing secret ever has to travel. This is public key encryption, and it exists to solve exactly the key-sharing problem symmetric encryption can’t.

real systems use both

Symmetric is fast but can’t safely share its key; asymmetric can share keys safely but is slow. So HTTPS uses asymmetric encryption once at the start — just long enough to secretly agree on a fresh symmetric key — then switches to fast symmetric encryption for the rest of the conversation. Best of both. (More in Section 09.)

The genius answer — public key encryption — uses a pair of mathematically linked keys:

🔓 Public key — share with everyone

Like a pile of open padlocks Bob hands out freely. Anyone can snap one shut on a box (encrypt a message to Bob). But snapping a padlock shut doesn't help you open one.

🔑 Private key — never leaves Bob

The only key that opens those padlocks (decrypts). Bob keeps it secret forever. What the public key locks, only the private key unlocks.

why can't Eve just reverse it?

The key pair is built from math that's easy one way, brutally hard backwards. Quick: what's 61 × 53? You can do that on paper (3233). Now: which two prime numbers multiply to make 9797? Suddenly you're guessing. Real systems use primes hundreds of digits long — multiplying them takes a computer microseconds; un-multiplying (factoring) would take longer than the age of the universe. Eve has the public key and the scrambled message, and it still doesn't help her.

big idea, exam version

Public key encryption means two parties can communicate securely without ever meeting to exchange a secret key — the public key can be shared openly because it only encrypts; only the paired private key decrypts.

09The padlock in your browser

HTTP vs. HTTPS

HTTP (HyperText Transfer Protocol) is the protocol of the Web — the rules a browser uses to request pages and a server uses to respond. It works great. It has one catastrophic flaw: everything travels as plain readable text.

HTTPS is the same HTTP with one added letter — S for Secure — wrapping every request in encryption (a system called TLS). And here's the connection to Section 08: when you visit an HTTPS site, your browser uses the site's public key to secretly agree on encryption for the whole conversation. Public key cryptography is happening every time you see the padlock.

What Eve sees on the wire

You're logging into a website while Eve watches the network traffic (easy to do on public Wi-Fi). Flip the protocol switch and send the login.

http://sketchy-games.com/login
HTTP ← flip me
📡 Eve's packet sniffer waiting for traffic…
the padlock means less than you think

HTTPS guarantees the connection is encrypted in transit — Eve can't read it. It does not guarantee the website itself is honest. A scam site can have a perfectly valid padlock. HTTPS answers "can anyone snoop on this conversation?" — not "should I trust who I'm talking to?"

Cheat Sheet

the internet — quick reference
every term from this page, one line each
Internet
Global network of networks — hardware + open protocols that move any data.
World Wide Web
Pages, files & links (information) accessed over the Internet via HTTP.

Unique address per device. IPv4 = 4 bytes (32 bits), each 0–255. IPv6 = 128 bits.

DNS
The Internet’s phone book — translates a domain name into an IP address. A subdomain sits to the left of its domain.
IP / TCP
IP = addressing & routing (no promises). TCP = order, delivery confirmation, reassembly.
TCP vs. UDP
TCP = reliable (re-sends, in order): web, email, files. UDP = on-time (no re-sends): calls, streaming, games.
Packet
Small numbered chunk of a message + destination address; each routes independently.
Redundancy
Extra paths beyond the minimum — the reason failures don't stop delivery.
Fault tolerance
System keeps working when parts fail; packets route around damage.
Scalability
Grows without redesign — open protocols, no central computer, hierarchy.
Bandwidth
Maximum capacity of a connection (Mbps). Lanes on the highway.
Bit rate
Bits actually transferred per second — can't exceed bandwidth.
Latency
Delay (ms) for a round trip. Gaming & calls live and die by it.
Transmission media
Fiber = fastest/most over distance (light). Copper = cheap, shorter reach. Radio/5G = wireless, shared. Satellite = far reach, high latency.
Digital divide
Gap in access to computing/Internet across geography, income, infrastructure.
Lossless
Compress via patterns; decompression restores the original exactly. Text, code, ZIP, PNG.
Lossy
Deletes detail humans barely notice; unrecoverable but much smaller. JPEG, MP3, video.

One shared secret key locks & unlocks — fast, but the key must be shared safely first. HTTPS uses it after an asymmetric handshake.

Shared openly; encrypts only. An open padlock anyone can snap shut.
Private key

HTTPS
HTTP + encryption (TLS), set up using public-key crypto. Padlock = private, not trustworthy.
