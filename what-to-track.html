Deciding what to track

The hard part of an array problem usually isn't the loop — it's figuring out what to keep track of as you walk through it. Here's a repeatable way to decide, worked through four classic CodingBat problems.

The method

Four questions that pick your variables for you

Every problem below cracks open with the same four questions. Step 2 is where the real decision lives — that's the one to slow down on.

1
Name the goal

What does the function hand back, and what does that value mean in plain English? "A boolean — true when there are more 1's than 4's." "An int — the gap between biggest and smallest."

2
Ask: what would I need to KNOW to answer that?

Turn the goal into the facts you'd need about the whole array. Each fact becomes one variable you track. "More 1's than 4's" needs how many 1's and how many 4's — two counters. This step is the whole game.

3
Set up each tracker

Give every tracked variable a starting value and a one-line update rule. Counters and sums start at 0; a running max or min starts at the first element (never 0 — remember the negatives trap). Update it each pass using nums[i].

4
Combine into the answer

After the loop, your tracked values hold everything you need. Here you'd check whether count1 > count4 and return true or false accordingly.

The one habit that matters ↓Before writing any loop, finish this sentence out loud: "To answer this, I need to know ___ and ___." Every blank you name is a variable to declare before the loop. If you can't fill the blanks yet, you're not ready to write the loop — and that's fine. That is the thinking part, and it's the part they're testing.
Problem 1

more14 — counting two things at once

Return true if the array holds more 1's than 4's.

public boolean more14(int[] nums)
[1, 4, 1, 4, 1]→true [1, 4, 4]→false [4, 4, 1]→false
decide first →What does more14 need to keep track of?

Walking it through

Goal: a boolean — "are there more 1's than 4's?" What I need to know: the count of 1's and the count of 4's — two counters, both starting at 0. Update: each pass, if the value is 1 bump count1; if it's 4 bump count4. Combine: return whether count1 beats count4.

public boolean more14(int[] nums) {
  int count1 = 0;                  // track: how many 1's
  int count4 = 0;                  // track: how many 4's
  for (int i = 0; i < nums.length; i++) {
    if (nums[i] == 1) {
      count1 = count1 + 1;
    }
    if (nums[i] == 4) {
      count4 = count4 + 1;
    }
  }
  if (count1 > count4) {           // combine the two trackers
    return true;
  } else {
    return false;
  }
}

Two separate ifs, not else if — though a value can't be both 1 and 4, so it wouldn't matter here. Two independent counters is the move whenever a problem compares the frequency of two things.

For later — the short way ↓Once true/false really clicks, that whole if/else collapses into a single line: return count1 > count4;. The comparison already is a true/false value, so you can hand it straight back. It's a fine shortcut when you want it — but the if/else above is just as correct, and easier to read while it's still settling in. (Same trick works for sum28: return sum2 == 8;.)
Problem 2

sum28 — one tracker, but which one?

Add up only the 2's in the array. Return true if that total comes to exactly 8 — which is really asking "are there exactly four 2's?"

public boolean sum28(int[] nums)
[2, 3, 2, 2, 4, 2]→true [2, 3, 2, 4]→false [2, 2, 2, 2]→true
decide first →What does sum28 need to keep track of?

Walking it through

Goal: a boolean — "do the 2's add up to 8?" What I need to know: just the running total of the 2's — one accumulator at 0. Update: each time the value is 2, add 2 to it. Combine: return whether that total equals 8.

public boolean sum28(int[] nums) {
  int sum2 = 0;                    // track: the running total of the 2's
  for (int i = 0; i < nums.length; i++) {
    if (nums[i] == 2) {
      sum2 = sum2 + 2;             // each 2 adds 2 to the total
    }
  }
  if (sum2 == 8) {                 // combine: is that total exactly 8?
    return true;
  } else {
    return false;
  }
}
Read the goal twice ↓"The sum of the 2's is 8" means each 2 contributes 2 toward a total, so four 2's. It does not mean the array contains the number 8. Misreading the goal in step 1 is exactly how you end up tracking the wrong thing in step 2.
Problem 3

bigDiff — two trackers, careful starting values

The array has at least one number. Return the gap between its largest and smallest values.

public int bigDiff(int[] nums)
[10, 3, 5, 6]→7 [7, 2, 10, 9]→8 [2, 10, 7, 2]→8
decide first →What does bigDiff need to keep track of?

Walking it through

Goal: an int — the spread, biggest minus smallest. What I need to know: the max and the min. Both start at nums[0] (a real value from the array). Update: bigger than max → new max; smaller than min → new min. Combine: max - min.

public int bigDiff(int[] nums) {
  int max = nums[0];               // track: biggest so far (start at first)
  int min = nums[0];               // track: smallest so far (start at first)
  for (int i = 0; i < nums.length; i++) {
    if (nums[i] > max) {
      max = nums[i];
    }
    if (nums[i] < min) {
      min = nums[i];
    }
  }
  return max - min;                // combine: the spread
}
The trap that bites here ↓Starting int max = 0; feels natural — and it's wrong. If every value is negative, 0 would never get beaten and you'd return a max that isn't even in the array. Start both at nums[0] so your champion is always a real value. (Shortcut: Processing's own max(max, nums[i]) and min(min, nums[i]) can replace the if-blocks once you trust the pattern.)
Problem 4 · boss level

centeredAverage — three trackers working together

Return the average of the array after dropping one copy of the largest value and one copy of the smallest. At least 3 numbers, and use integer division.

public int centeredAverage(int[] nums)
[1, 2, 3, 4, 100]→3 [1, 1, 5, 5, 10, 8, 7]→5
decide first →What does centeredAverage need to keep track of?

Walking it through

This is the earlier problems stacked together: you need the total (like sum28) and the max and min (like bigDiff). Track all three in one loop. To "ignore the largest and smallest," subtract one max and one min from the sum. You removed exactly two values, so divide by nums.length - 2.

public int centeredAverage(int[] nums) {
  int sum = 0;                     // track: total of everything
  int max = nums[0];               // track: biggest so far
  int min = nums[0];               // track: smallest so far
  for (int i = 0; i < nums.length; i++) {
    sum = sum + nums[i];
    if (nums[i] > max) {
      max = nums[i];
    }
    if (nums[i] < min) {
      min = nums[i];
    }
  }
  return (sum - max - min) / (nums.length - 2);   // drop one max & one min
}
Why this just works ↓Subtracting one max and one min removes exactly one copy of each extreme — so even if the smallest value appears three times, only a single copy leaves the sum (that's the "ignore just one copy" rule, handled for free). Dividing by length - 2 matches the two values you pulled out. Integer division drops the remainder, which is what the problem asks for.

sum67 — now you decide

Same method, fresh problem — and this time you write the loop. Add up the numbers, but ignore every stretch that starts with a 6 and runs through the next 7 (the 6 and the 7 are both skipped; every 6 has a 7 somewhere after it). An empty array returns 0.

public int sum67(int[] nums)
[1, 2, 2]→5 [1, 2, 2, 6, 99, 99, 7]→5 [1, 1, 6, 7, 2]→4
decide first →Before writing a line — what does sum67 need to keep track of?

Now run the method yourself

1 — Goal: an int, the sum of the numbers you keep. 2 — What to know: the running sum, plus a skip flag that's true while you're between a 6 and its 7. 3 — Set up: int sum = 0; and boolean skip = false;, then decide for each element whether to flip the flag or add the value. 4 — Combine: just return sum;.

Light hint, then go ↓The order you check things matters. Both the 6 and its closing 7 get skipped — so handle "am I starting or ending a skip zone?" before you decide whether to add the number. Trace [1, 1, 6, 7, 2] by hand first, watching skip flip on at the 6 and off at the 7. The code is yours to write.
The bigger lesson ↓Look back at all five: counters, a sum, the two extremes, three trackers at once, and now a mode flag. The thing you track is never the same — but the way you find it always is. Name the goal, fill in "I need to know ___," and your variables fall out of the answer.
The takeaway

Same method, four different answers

Notice what actually changed from problem to problem: not the loop, not the four-step process — only what you decided to track. And that decision fell straight out of naming the goal.
