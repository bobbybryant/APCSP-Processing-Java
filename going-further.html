Add-on 1A pop-up menu instead of typing

Typing the word steel works, but a button menu is friendlier and can't be mistyped. Swap the typed prompt for showOptionDialog, which hands back the index of whichever button the rider clicked.

Building blocks: page 1 → A pop-up multiple choice
void setup() {
  String[] options = { "steel", "wooden" };
  int choice = JOptionPane.showOptionDialog(null,
      "What type of coaster do you want?", "Coaster Picker",
      0, 3, null, options, options[0]);
  // choice is the button index: 0 = steel, 1 = wooden

  if (choice == 0) {
    findCoastersToRide(false);
  } else {
    findCoastersToRide(true);
  }
}

A nice side effect: the typed version sent anything that wasn't "steel" down the wooden path (even a typo). Two buttons remove that guesswork — the menu only ever offers valid choices.

Add-on 2Let the user add their own coasters

Right now the four coasters are hard-coded. Let the rider add their own: ask for a name with a text box, then ask steel or wooden with a button menu, and append to the lists. The move that keeps everything working — append to both parallel lists so index x still lines up.

Building blocks: text input + the button menu (page 1)
void addCoaster() {
  String name = JOptionPane.showInputDialog("Name of the coaster?");

  String[] types = { "steel", "wooden" };
  int t = JOptionPane.showOptionDialog(null,
      "Is " + name + " steel or wooden?", "Coaster Type",
      0, 3, null, types, types[0]);

  coasters.append(name);     // grow the name list...
  if (t == 1) {              // wooden button
    wooden.append(1);        // ...and grow the wooden list to match
  } else {                   // steel button
    wooden.append(0);
  }
}

This is exactly the situation 2025 Q2C warned about: add to the list and the parallel list has to grow too. Here we do it on purpose, every single time.

Add-on 3Track height too

Want more than names and types? Add a third parallel list for height. Declare it next to the others, and prompt for a height when adding a coaster (with int() to turn the typed text into a number).

Building blocks: text input + int() cast (page 1)
IntList heights = new IntList(205, 110, 125, 170);   // feet, parallel to coasters

Then the addCoaster procedure grows one more prompt and one more append:

void addCoaster() {
  String name = JOptionPane.showInputDialog("Name of the coaster?");

  String[] types = { "steel", "wooden" };
  int t = JOptionPane.showOptionDialog(null,
      "Is " + name + " steel or wooden?", "Coaster Type",
      0, 3, null, types, types[0]);

  String h = JOptionPane.showInputDialog("How tall is " + name + " (in feet)?");

  coasters.append(name);          // all three lists...
  heights.append(int(h));         // ...grow together...
  if (t == 1) {
    wooden.append(1);             // ...so index x still lines up everywhere
  } else {
    wooden.append(0);
  }
}

Now three lists move together. The rule just scales: one append per list, every time.

Add-on 4Check the rider's thrill level

Heights unlock something fun: a thrill level. Write one small helper whose only job is turning a height into a tier — then ask the rider how much thrill they want and keep only the coasters at or below it.

Building blocks: the button menu + your own helper procedure (a height → tier function, one job each — just how functions should work)
String[] thrillNames = { "mild", "moderate", "extreme" };

int thrillRank(int feet) {       // one job: turn a height into a thrill tier
  if (feet > 200) {
    return 2;                    // extreme
  } else if (feet >= 130) {
    return 1;                    // moderate
  } else {
    return 0;                    // mild
  }
}

Then findCoastersToRide asks for a thrill level and adds it to the test. The type checks from page 3 are untouched — we just tack on one more && and print each coaster's tier:

void findCoastersToRide(boolean madeOfWood) {
  String[] levels = { "mild", "moderate", "extreme" };
  int pick = JOptionPane.showOptionDialog(null,
      "How much thrill can you handle?", "Thrill Level",
      0, 3, null, levels, levels[0]);

  for (int x = 0; x < coasters.size(); x++) {
    if (wooden.get(x) == 1 && madeOfWood == true && thrillRank(heights.get(x)) <= pick) {
      println("The coaster " + coasters.get(x) + " is wooden (" + thrillNames[thrillRank(heights.get(x))] + ").");
    } else if (wooden.get(x) == 0 && thrillRank(heights.get(x)) <= pick) {
      println("The coaster " + coasters.get(x) + " is steel (" + thrillNames[thrillRank(heights.get(x))] + ").");
    }
  }
}

Add-on 5A strict-filter switch

By default, asking for wooden also lists the steel coasters — on purpose. A rider happy to take the rougher wooden coasters will gladly ride the smoother steel ones, so "wooden" really means "anything I can handle." But a rider who specifically wants one type might prefer an exact match. Give them the choice.

Strict behavior is a single extra check on the steel branch — require madeOfWood == false so steel only shows on a steel request:

} else if (wooden.get(x) == 0 && madeOfWood == false) {

To make it optional, add a second boolean and let it decide. Steel shows when you asked for steel or you're not being strict:

void findCoastersToRide(boolean madeOfWood, boolean strictType) {
  for (int x = 0; x < coasters.size(); x++) {
    if (wooden.get(x) == 1 && madeOfWood == true) {
      println("The coaster " + coasters.get(x) + " is wooden.");
    } else if (wooden.get(x) == 0 && (madeOfWood == false || strictType == false)) {
      // steel shows when you asked for steel, OR when you aren't filtering strictly
      println("The coaster " + coasters.get(x) + " is steel.");
    }
  }
}
Building blocks: a yes/no confirm dialog feeds the switch (page 1)

…and setup asks the question, then passes the answer along:

int strict = JOptionPane.showConfirmDialog(null,
    "Show only that type? (No = show everything I can ride)", "Strict filter?",
    JOptionPane.YES_NO_OPTION);
boolean strictType = (strict == JOptionPane.YES_OPTION);   // Yes -> strict

if (choice == 0) {
  findCoastersToRide(false, strictType);
} else {
  findCoastersToRide(true, strictType);

Keep asking until the input makes sense

A while loop re-prompts until the height is a real, positive number. (A blank or letters cast to 0, so this catches those too.)

Building blocks: keep asking until correct (page 1)
String h = JOptionPane.showInputDialog("How tall is " + name + " (in feet)?");
while (int(h) <= 0) {            // keep asking until it's a real, positive number
  h = JOptionPane.showInputDialog("Please enter a height greater than 0:");
}
heights.append(int(h));

A quick summary at the end

Reuse the array patterns: the count pattern tallies the wooden coasters, and a findMax loop reports the tallest one.

Building blocks: the array patterns (page 1)
// how many wooden coasters are there? (the "count" pattern)
int woodenCount = 0;
for (int x = 0; x < wooden.size(); x++) {
  if (wooden.get(x) == 1) {
    woodenCount++;
  }
}
println("There are " + woodenCount + " wooden coasters.");

// which coaster is tallest? (the "find an extreme" pattern)
int tallest = heights.get(0);
String tallestName = coasters.get(0);
for (int x = 0; x < heights.size(); x++) {
  if (heights.get(x) > tallest) {
    tallest = heights.get(x);
    tallestName = coasters.get(x);   // keep the name in step with the height
  }
}
println("Tallest coaster: " + tallestName + " at " + tallest + " ft");

The point

Every feature on this page is optional. The version on page 3 already met every Create Task requirement before any of this existed — and that's the whole lesson:

  1. Meet the requirements first. A list, a procedure with a used parameter, sequencing/selection/iteration, a call, input, output. Get it working and saved.
  2. Then, with time to spare, layer on extras — menus, user input, new data, helpers.
  3. Chase features before the core works and you can run out of time with nothing that scores.
