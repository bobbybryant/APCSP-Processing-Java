04Base 16

Hexadecimal

Hex is base 16: digits run 0–9, then A–F (A=10, B=11, C=12, D=13, E=14, F=15). The killer feature: one hex digit is exactly four binary bits (a “nibble”). So converting hex ↔ binary is just swapping each digit for its 4-bit pattern — no math required.

why hex exists (exam answer!)

Binary is base 2 — what computers actually use to store and process data. Hex is base 16 — a compact way for humans to read and write binary, common for things like memory addresses and web color codes (#FF6B35). Hex is not a separate language computers “speak” — it's shorthand for binary.

05Patterns worth knowing

Number Tricks & Powers of 2

Two classic short-answer questions hide in here, plus the “how many values fit in n bits” math you need everywhere else.

➡️ The append-a-zero trick

Adding a 0 to the rightmost place shifts every digit into a place worth one base more. In binary that doubles the value (×2); in decimal it multiplies by 10. A 0 on the leftmost end? Changes nothing — leading zeros have no value.

📊 How much fits in n bits?

Each extra bit doubles the number of patterns. With n bits you get 2ⁿ values, and since counting starts at 0, the biggest value is 2ⁿ − 1. Drag the slider:

06Bits meet pixels

RGB Color Math

A computer mixes every color from red, green, and blue light. Each channel gets 1 byte (8 bits), so each runs 0–255. That's 256 choices, three times:

256 × 256 × 256 = 2²⁴ = 16,777,216 colors

🎨 The channel mixer
R232
G114
B90
writing the exam answer

The full-credit explanation hits three beats: (1) each of R, G, B is stored in 8 bits, so each has 256 possible levels; (2) the channels combine, so multiply: 256 × 256 × 256; (3) that's 2²⁴ ≈ 16.7 million colors. Notice it's the same “2ⁿ values” rule from the last section — with n = 24.

The exam loves two skills here: choosing the most efficient type (fewest bits that still fit the value), and knowing which naming rules are actual rules versus just good style.

"most efficient" decoded

Efficient = the smallest type that can still hold the value. 117 fits in a byte — using an int works, but wastes 3 bytes. A decimal like 513.17? Smallest decimal type: float. Text like "B+" has more than one character, so it's a String, but a lone 'K' is a char. A number in the billions blows past int's range → long.

Naming rules: required vs. style

Hard rules (code breaks)

• One word — no spaces
• Letters, digits, and _underscore only — no other punctuation
• Must begin with a letter (or underscore) — digits allowed after that
• Can't be a reserved word — including Processing's own names like width, height, mouseX, mouseY, color, size

Style conventions (code still runs)

• camelCaseCapitalization — strongly encouraged, not required
• Descriptive names like playerScore instead of ps

Watch for the trap: “may not contain any numbers at all” is FALSE — numbers just can't come first.

08When types run out

When Numbers Overflow

Every numeric type has a maximum value baked in by its number of bits. Push a value past that ceiling and it wraps around like a car odometer rolling past 999,999 — the stored number suddenly looks tiny or, in Java's signed integers, negative.

🚶 The step-counter experiment

A fitness tracker stores steps in 2 bytes (max 65,535). Keep walking and see what happens at the ceiling.

0
The counter has 65,536 possible values: 0 through 65,535.
spot it on the exam

Story problems love this bug: a counter (views, scores, steps, likes) climbs for a while, then suddenly displays a negative or nonsense number while everything else works fine. The answer is almost always: the value exceeded the maximum its data type can store. It's not a rounding error, not server overload, not a sampling problem — the type simply ran out of bits.

09Write once, use forever

Functions

A function is a named, reusable block of code. Two completely different moments in its life:

Defining = writing the recipe

You describe what the function does — its name, its parameters, and the code inside. Nothing runs yet. A definition sitting in your sketch does nothing on its own.

void drawBalloon(int x, int y, int d, color c) {
  fill(c);
  ellipse(x, y, d, d);          // the balloon
  line(x, y + d/2, x, y + d);   // the string
}
Calling = cooking from it

You use the function's name with real values (arguments). This is when the code inside actually runs — once per call, with the values you passed in.

drawBalloon(120, 80, 60, red);
drawBalloon(300, 140, 90, teal);
drawBalloon(420, 60, 40, gold);
syntax check: parameters

In a definition, every parameter needs its own type, and parameters are separated by commas: void mix(int a, int b, int c). Missing commas or missing types = broken definition. The exam will show you near-miss versions — look for both.

🎈 The function machine

drawBalloon() is defined once above. Change the arguments and call it as many times as you want — that's the whole point of parameters.

x y d
Your calls will appear here, exactly as code.
"why bother?" — the exam answer

Functions cost time up front but pay it back fast: write once, call many times (no copy-paste); one place to fix bugs (change the definition, every call updates); readable code (a clear name like drawBalloon says what it does); and easier teamwork — someone can use your function without reading its insides.

10First true wins

Conditionals

An if / else if / else chain checks conditions top to bottom and runs only the first one that's true — then skips the rest. That single fact explains the most common exam bug: a ladder with conditions in the wrong order.

🪛 The ladder, ordered vs. scrambled

A theme park prices tickets by age. Drag the age slider and watch which branch fires in each version.

age 15
✓ Correct: biggest threshold first
✗ Broken: smallest threshold first
the fix, in one sentence

With >= ladders, check from the highest threshold down (or use ranges). If a small threshold like age >= 13 sits on top, it “catches” every adult and senior before their branches ever get checked. Also: all else if branches must come before the final else — an else if after else is a syntax error.

Two decisions? Two separate ifs.

When choices are independent — like a base price plus an optional add-on — nesting one inside the other accidentally locks the add-on to one group. Compare:

✓ Separate ifs: everyone can add the meal plan 
int cost = 30;            // child base price
if (age >= 16) {
  cost = cost + 15;       // adult upgrade
}
if (wantsMeal == true) {
  cost = cost + 12;       // ANYONE can add this
}
println(cost);
✗ Nested: kids can never buy the meal plan 
int cost = 30;
if (age >= 16) {
  cost = cost + 15;
  if (wantsMeal == true) {
    cost = cost + 12;     // only runs for 16+ !
  }
}
println(cost);

Predict the output

Cover the answer, trace it on paper, then reveal. The skill: substitute the values, evaluate the condition, follow exactly one branch.

int x = 7;
int y = 12;
if (x > y) {
  println(x * y);
} else {
  println(y - x);
}

5. 7 > 12 is false, so the else branch runs: 12 - 7 = 5. The multiply never happens.

int a = 4;
int b = 9;
if (a > 2 && b < 9) {
  println("alpha");
} else if (a > 2 || b < 9) {
  println("beta");
} else {
  println("gamma");
}

beta. First condition: a > 2 is true but b < 9 is false — && needs both, so it fails. Second condition: || needs just one, and a > 2 is true → “beta” prints and the chain stops.

int temp = 65;
if (temp >= 90) {
  println("hot");
} else if (temp >= 60) {
  println("mild");
} else if (temp >= 32) {
  println("chilly");
} else {
  println("freezing");
}

mild. 65 fails the first check, passes temp >= 60 — and even though 65 >= 32 is also true, that branch never runs. First true wins; the rest are skipped.

11Predict the canvas

Drawing in Processing

The exam shows code and a picture and asks you to match them. Four things to verify, in order:

1 · size(width, height)

Width first, height second. size(150, 300) is a tall window; size(300, 150) is a wide one. This alone eliminates half the answer choices.

2 · background(...)

One number = grayscale: background(0) is black, background(255) is white. It repaints the entire canvas, covering anything drawn before it.

3 · fill(r, g, b) comes BEFORE the shape

Fill is like dipping the brush: it colors shapes drawn after it. fill(255,0,0) = red, fill(0,0,255) = blue. A fill after the shape arrives too late.

4 · Coordinates from the top-left

(0,0) is the top-left corner; y grows downward. circle(x, y, d) centers at (x, y) with diameter d — so circle(50, 50, 100) hugs the top-left.

  12Big ideas about data
  
Compression & Metadata

  
Two vocabulary-driven topics that show up as story problems. Nail the definitions and the questions answer themselves. (For the interactive deep dive, revisit the Internet page's compression squeezer.)


  

    

      Lossless compression
      
Stores the same information more cleverly. Decompressing rebuilds the original bit-for-bit, exactly.

      
Trade-off: smaller space savings. Use when every detail matters: code, documents, spreadsheets.

    

    

      Lossy compression
      
Permanently throws away detail humans barely notice. The original can never be fully restored.

      
Trade-off: much smaller files → faster transmission. Use when “close enough” is fine: streaming video, photos, music.

    

  


  

    two facts the exam tests directly
    
(1) Lossy generally shrinks files more than lossless — so it cuts transmission time more. (2) “Lossless” describes round-trip behavior: compress → send → restore to the exact original → use. If a scenario never compresses anything, it isn't demonstrating compression at all!

  


  
Data vs. metadata

  
Data is the content itself — the message, the photo, the video. Metadata is data about the data — when it was posted, who posted it, how many comments it got. Rule of thumb: questions about what the content says need the data; questions about who / when / how often need only the metadata.

  14Write real code
  
Free-Response Practice

  
The midterm ends with writing functions from scratch. Practice on paper first — no autocomplete on test day — then reveal a full solution to compare against.


  

    Challenge A: a boolean function
    
In a game, a player levels up when they have enough experience and enough coins — or if they're holding a golden key, which skips the requirements entirely. Write a function called canLevelUp that takes two integers, xp and coins, and a boolean hasKey. Return true if xp is at least 100 and coins is at least 50, or if hasKey is true. Return false otherwise.

    
canLevelUp(120, 60, false)  -> true   // met both requirements
canLevelUp(120, 10, false)  -> false  // not enough coins
canLevelUp(0,   0,  true)   -> true   // the key overrides everything

    
    

      
boolean canLevelUp(int xp, int coins, boolean hasKey) {
  if ((xp >= 100 && coins >= 50) || hasKey == true) {
    return true;
  } else {
    return false;
  }
}

      
Graders look for: return type boolean (not void!), three typed parameters — including a boolean one — with commas, the two requirements joined by && and then OR’d with the key using ||, and a return on every path. Once you’re comfortable, the whole body can shrink to one line: return (xp >= 100 && coins >= 50) || hasKey;

    

  


  

    Challenge B: two functions working together
    
You're building one bar of a bar chart.

    1. Write calculateBarHeight: it takes a data value and how many pixels tall each unit should be, multiplies them, stores the result in a variable, and returns it.

    2. Write drawDataBar with five parameters for the bar's position, width, its data value, and whether to label it. It calls calculateBarHeight to get the height, draws the bar growing upward from a baseline, and displays the data value just above the bar when appropriate.

    

      the twist
      
Bars grow up, but y increases down the screen. So a bar sitting on baseline baseY has its top-left corner at baseY - barHeight — that's the value you pass to rect().

    

    
    

      
int calculateBarHeight(int dataValue, int pixelsPerUnit) {
  int barHeight = dataValue * pixelsPerUnit;
  return barHeight;
}

void drawDataBar(int x, int baseY, int barWidth, int dataValue, boolean showValue) {
  int barHeight = calculateBarHeight(dataValue, 4);
  rect(x, baseY - barHeight, barWidth, barHeight);
  if (showValue == true) {
    text(dataValue, x, baseY - barHeight - 5);
  }
}

      
Key moves: the calculator returns a value instead of printing it; the drawer's fifth parameter is a boolean; the call's result is caught in a variable; the upward growth comes from subtracting the height from the baseline (baseY - barHeight); and the if gates only the label — not the bar. Notice the call passes a literal 4 as an argument — arguments can be plain values, not just variables.

    

  


  

    free-response checklist
    
Before you move on, scan your function for the big five: return type correct? Every parameter typed, with commas? Curly braces paired? Semicolons on statements? If it returns something — is there a return on every path through the code?
