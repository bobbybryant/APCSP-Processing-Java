Worked example: RollerCoasterFilter

One tiny program, taken apart every way the exam can ask. Below is a complete Create Task-style program, then model answers to all four 2025 prompts and all four 2026 prompts — each with the reasoning behind it. This is a worked example to learn the moves from; your own project is still your own.

01

The example program

RollerCoasterFilter asks whether you want steel or wooden coasters, then prints the ones you can ride. It is deliberately small — readable in a minute — yet it still contains every required Create Task ingredient, which is exactly what makes it easy to point at when the prompts ask about your code.

import javax.swing.JOptionPane;

StringList coasters = new StringList("Millenium Force", "The Beast", "Gemini", "Gatekeeper");
IntList wooden = new IntList(0, 1, 1, 0);
StringList wantToRide = new StringList();

void findCoasters(boolean madeOfWood) {
  for (int i = 0; i < coasters.size(); i++) {
    if (madeOfWood == true && wooden.get(i) == 1) {
      wantToRide.append(coasters.get(i));
    } else if (madeOfWood == false && wooden.get(i) == 0) {
      wantToRide.append(coasters.get(i));
    }
  }

  println("Out of the available coasters, you can ride:");
  for (int i = 0; i < wantToRide.size(); i++) {
    println((i + 1) + ": " + wantToRide.get(i));
  }
}

void setup() {
  String input = JOptionPane.showInputDialog("What type of coaster do you want? Enter 1 for steel, or 2 for wooden");
  int ans = int(input);
  if (ans == 1) {
    findCoasters(false);
  } else if (ans == 2) {
    findCoasters(true);
  } else {
    JOptionPane.showMessageDialog(null, "That wasn't a valid input. Goodbye!");
  }
}

Try it

Pick a type and watch what the program prints. (Wooden: The Beast, Gemini. Steel: Millenium Force, Gatekeeper.)

02

Why it's built this way

Every design choice maps to a requirement. Here is the tour.

A list — and a parallel list

coasters holds the names; wooden holds a matching 0/1 for each one (1 = wooden). They are parallel lists: index i in each is about the same coaster. That satisfies the collection requirement and keeps related data tidy without a separate variable per coaster.

A procedure with a parameter that actually matters

findCoasters(boolean madeOfWood) is the student-developed procedure. The parameter madeOfWood isn't decoration — it decides which coasters qualify, so the same procedure produces two different results depending on what you pass in.

Sequencing, selection, and iteration — all inside it

Inside findCoasters: it runs steps in order (filter, then print a header, then print results) — that's sequencing; it chooses with if / else if — that's selection; and it walks the lists with for loops — that's iteration. One procedure covers all three.

Calling it, driven by input

setup() asks the user for 1 or 2, converts the text with int(), and calls findCoasters(false) or findCoasters(true). The else handles anything else, so a stray input doesn't break the program.

Input in, output out

JOptionPane gathers the input; println produces the visible output. Both are easy to talk about in a written response.

the takeawaySmall and clear beats big and tangled. Because each requirement lives in an obvious spot, every prompt below has a clean place to point.

2025 written response

Four prompts, answered for this program. The 2025 set is in blue.

2025 · 1Example output
the prompt, in shortGive one example of your program's output and explain what part of the program's functionality it shows.
answer

If the user enters 2, the program calls findCoasters(true) and prints:

Out of the available coasters, you can ride: 1: The Beast 2: Gemini

This shows the program's core feature: filtering the list by coaster type. The two coasters printed are exactly the ones whose wooden value is 1 (The Beast and Gemini), so the output demonstrates that the selection inside findCoasters kept only wooden coasters as the loop scanned the list.

why this answerQ1 wants a specific output plus a tie to a named feature. So I fix a concrete input (2), show the exact text it prints, and explain why those two — pointing at the filtering behavior rather than saying a vague "it lists some coasters."
2025 · 2AThe Boolean expression
the prompt, in shortLook at the first selection (if) statement in your procedure. Identify its Boolean expression, give specific value(s) that make it evaluate to true, and explain why.
answer

The first selection statement is the opening if inside findCoasters. Its Boolean expression is:

madeOfWood == true && wooden.get(i) == 1

It is true when madeOfWood is true and wooden.get(i) is 1. For example, the user enters 2 (so madeOfWood is true) and i is 1, where wooden.get(1) is 1 (The Beast). Both sides are true, so the whole expression is true and that coaster is added to wantToRide.

why this answerThe operator is && (logical AND), which is only true when both sides are true. So a complete answer has to pin down a value for the parameter and an index where wooden is 1 — then explain that the AND needs both.
2025 · 2BIntroducing a logic error
the prompt, in shortDescribe a change another programmer could make inside your procedure that would create a logic error, and describe how the procedure's behavior would change.
answer

Change the else if test from == true to == false:

} else if (madeOfWood == false && wooden.get(i) == 1) {   // was: == true

Now both branches only fire for wooden coasters. When the user asks for steel (madeOfWood is false), the first branch is skipped, but the else if — madeOfWood == false && wooden.get(i) == 1 — now matches the wooden coasters. So a steel request would wrongly print The Beast and Gemini, and the steel coasters would never appear. The program still runs without crashing; it just produces the wrong list.

why this answerA logic error runs fine but gives wrong output — so I pick a one-character condition change that compiles and runs. I deliberately avoid something like changing < to <=, because that crashes with an out-of-bounds error — that's a runtime error, not a logic error.
2025 · 2CIf the list grows
the prompt, in shortSuppose another programmer adds several elements to the end of the list. Explain how the part (ii) code segment would need to change — or, if no change is needed, explain why.
answer

No change is needed. The loop is bounded by coasters.size(), not a fixed number like 4. .size() always reports the current number of elements, so adding coasters to the end just makes the loop run more times and process the new ones automatically.

One honest caveat: the parallel wooden list would need a matching 0 or 1 added for each new coaster so the indices still line up — but that's new data, not a change to the part (ii) code.

why this answerThis is the payoff of writing i < coasters.size() instead of i < 4. Because .size() adapts on its own, the answer is "no change," and the explanation is simply why: the loop bound tracks the list's real length.

2026 · 1Useful documentation
the prompt, in shortDescribe one piece of documentation you could include with your program, and explain how another programmer could use it to better understand a particular code segment.
answer

A header comment above findCoasters describing what it does, its parameter, and its hidden dependency:

// findCoasters(madeOfWood): scans the coasters list and adds every coaster
// whose type matches madeOfWood (true = wooden, false = steel) to the
// wantToRide list, then prints them. Needs the parallel "wooden" list to
// stay lined up with "coasters" (same length, same order).
void findCoasters(boolean madeOfWood) {

Reading just that comment, another programmer immediately learns three things they'd otherwise have to reverse-engineer: what madeOfWood means (true = wooden), what the procedure produces (it fills wantToRide and prints it), and the rule that wooden must stay parallel to coasters. That lets them call and modify it safely without tracing the whole loop.

why this answerQ1 wants a specific documentation artifact tied to a specific segment — not "comments are good." A procedure header comment is the natural choice, and I connect it to this code's real confusion points: the meaning of the boolean and the parallel-list dependency.
2026 · 2AWhen the loop stops
the prompt, in shortFor the first iteration statement in your procedure, identify the variable(s) that determine when it stops, give specific values that stop it, and explain why. If there are no such variables, explain how it stops.
answer

The first iteration statement is the opening for loop:

for (int i = 0; i < coasters.size(); i++)

The values that control it are the counter i and coasters.size() (currently 4). The loop keeps going while i < coasters.size(). It stops when i reaches 4: at that point 4 < 4 is false, so the loop ends.

i starts at 0 and rises by 1 each pass, so it visits every index 0 through 3 exactly once before the test fails.

why this answerThe prompt asks for the controlling variable(s), an exact stopping value, and the reason. So I name both i and the .size() it's compared against, give i == 4 as the value where the condition flips false, and explain the counting.
2026 · 2BA test case that breaks it
the prompt, in shortWrite a call to your procedure with specific argument(s) that the procedure accepts but that makes it behave incorrectly; describe and explain the incorrect behavior. If that's not possible, explain why.
answer

For a single call, this is not possible, and explaining why earns the point. findCoasters takes one boolean, so the only arguments it accepts are true and false — and both are intended, handled values (true → wooden, false → steel). There is no "bad" boolean to pass, so no accepted single call produces wrong behavior.

an honest asideThe program does have a real weakness: wantToRide is a global that is never cleared, so calling findCoasters twice (say true then false) lets the second call's output still contain the first call's coasters. But that's caused by repeated calls and shared state, not by the argument of a single call — so it doesn't fit this prompt's "a call with specific argument(s)" framing.
why this answerWhen a parameter's entire domain is handled, the correct, rubric-friendly answer is the "not possible" branch — you earn credit by explaining that both boolean values are valid and handled by design. (I flag the global-state bug separately so the analysis stays honest, but it isn't what this specific prompt is asking for.)
2026 · 2CAbstraction & the list
the prompt, in shortExplain how the list uses abstraction to manage complexity. Then describe how you'd adjust the part (ii) code segment to keep the same behavior if the list were removed — or, if that isn't possible, explain why.
answer — the abstraction

One name, coasters, stands in for all four coaster names. coasters.get(i) and coasters.size() let a single loop handle any number of coasters, so the rest of the program never has to know how many there are or name them individually. The list hides that detail behind one name and an index — that's the abstraction.

answer — without the list

It is possible, but only by hard-coding a separate variable per coaster and giving up the loop. The part (ii) loop becomes a hand-written sequence — one if per coaster:

String coaster0 = "Millenium Force";  int wooden0 = 0;
String coaster1 = "The Beast";        int wooden1 = 1;
String coaster2 = "Gemini";           int wooden2 = 1;
String coaster3 = "Gatekeeper";       int wooden3 = 0;

// part (ii) loses its loop -- one hand-written if per coaster instead:
if (madeOfWood == true && wooden0 == 1) { wantToRide.append(coaster0); }
if (madeOfWood == true && wooden1 == 1) { wantToRide.append(coaster1); }
if (madeOfWood == true && wooden2 == 1) { wantToRide.append(coaster2); }
if (madeOfWood == true && wooden3 == 1) { wantToRide.append(coaster3); }
// ...plus the matching steel checks. Add a coaster -> add more lines by hand.

Same behavior, but every coaster you add now means more code written by hand. That repetition is exactly the complexity the list removes.

why this answer2C wants both halves: the abstraction explanation and a concrete without-list rewrite. So I explain "one name for many" plus get/size, then show the expanded version — individual variables and repeated ifs with no loop — to make the saved complexity visible.
