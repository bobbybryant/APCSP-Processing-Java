Worked example: RollerCoasterFilter

One small program, taken apart every way the exam can ask. Below is a complete Create Task-style program, then model answers to all four 2025 prompts and all four 2026 prompts — each with the reasoning behind it. Learn the moves here; your own project is still your own.

01

The example program

RollerCoasterFilter asks whether you want steel or wooden coasters, then prints each coaster, labeled by type. It is deliberately small — readable in a minute — yet it still contains every required Create Task ingredient, which is exactly what makes it easy to point at when the prompts ask about your code.

//Roller Coaster Filter
import javax.swing.JOptionPane;

StringList coasters = new StringList("Steel Vengeance", "Beast", "Gemini", "Gatekeeper");
IntList wooden = new IntList(0, 1, 1, 0);

void setup() {
  String input = JOptionPane.showInputDialog("Type steel or wooden for your coaster");
  if (input.toLowerCase().equals("steel")) {
    findCoastersToRide(false);
  } else {
    findCoastersToRide(true);
  }
}

void findCoastersToRide(boolean madeOfWood) {
  for (int x = 0; x < coasters.size(); x++) {
    if (wooden.get(x) == 1 && madeOfWood == true) {
      println("The coaster " + coasters.get(x) + " is wooden.");
    } else if (wooden.get(x) == 0) {
      println("The coaster " + coasters.get(x) + " is steel.");
    }
  }
}

Try it

Type a coaster type and run it. (Wooden coasters are Beast and Gemini; steel are Steel Vengeance and Gatekeeper.)

type:

Try both: type steel, then wooden. Notice that wooden lists every coaster while steel lists only steel — that asymmetry is intentional, and 2026 Q2B explains the reasoning.

02

Why it's built this way

Every required ingredient has an obvious home. Here is the tour.

A list — and a parallel list

coasters holds the names; wooden holds a matching 0/1 for each one (1 = wooden, 0 = steel). They are parallel lists: index x in each is about the same coaster. That covers the collection requirement and keeps related data tidy without a separate variable per coaster.

A procedure with a parameter that matters

findCoastersToRide(boolean madeOfWood) is the student-developed procedure. The parameter madeOfWood isn't decoration — it controls the first condition, so passing true vs false changes what the procedure does.

Sequencing, selection, and iteration — all inside it

Inside the procedure: statements run in order (sequencing), an if / else if chooses what to print (selection), and a for loop walks the whole list (iteration). One procedure covers all three.

a deliberate choiceLook closely at the second branch: else if (wooden.get(x) == 0) doesn't also check madeOfWood. That's on purpose. Asking for wooden coasters lists the steel ones too — because a rider happy on the rougher wooden coasters will certainly ride the smoother steel ones, so "wooden" really means "anything I can handle." Asking for steel stays steel-only. Page 4 adds an optional "strict" switch if you'd rather match the type exactly.

Calling it, driven by input

setup() asks the rider to type a type and compares it with input.toLowerCase().equals("steel") — the string-comparison move from the building blocks page. Type steel and it calls findCoastersToRide(false); anything else falls to the else and calls findCoastersToRide(true).

Input in, output out

JOptionPane gathers the typed input; println produces the visible output, one line per coaster.

the takeawaySmall and clear beats big and tangled. Because each requirement lives in an obvious spot, every prompt below has a clean place to point.

2025 written response

Four prompts, answered for this program. The 2025 set is in blue.

2025 · 1Example output
the prompt, in shortGive one example of your program's output and explain what part of the program's functionality it shows.
answer

If the rider types steel, the program calls findCoastersToRide(false) and prints:

The coaster Steel Vengeance is steel. The coaster Gatekeeper is steel.

This shows the program's core job: reading the parallel wooden list to label each coaster. The loop walks all four coasters, and for the steel ones (wooden.get(x) == 0) it prints the "is steel" line — so the output demonstrates the list traversal and the selection working together.

why this answerQ1 wants a specific output plus a tie to a named feature. So I fix a concrete input (steel), show the exact text it prints, and explain why those two coasters appear — pointing at the list read and the conditional, not a vague "it prints coasters."
2025 · 2AThe Boolean expression
the prompt, in shortLook at the first selection (if) statement in your procedure. Identify its Boolean expression, give specific value(s) that make it evaluate to true, and explain why.
answer

The first selection statement is the opening if inside findCoastersToRide. Its Boolean expression is:

wooden.get(x) == 1 && madeOfWood == true

It is true when wooden.get(x) is 1 and madeOfWood is true. For example, the rider types wooden (so madeOfWood is true) and x is 1, where wooden.get(1) is 1 (Beast). Both sides are true, so the whole expression is true and that coaster prints as wooden.

why this answerThe operator is && (logical AND), which is only true when both sides are true. So a complete answer pins down a value for the list element (wooden.get(x) = 1) and the parameter (madeOfWood = true), then explains that the AND needs both.
2025 · 2BIntroducing a logic error
the prompt, in shortDescribe a change another programmer could make inside your procedure that would create a logic error, and describe how the procedure's behavior would change.
answer

Change the && in the first condition to ||:

if (wooden.get(x) == 1 || madeOfWood == true) {   // was: &&

Now the first condition is true whenever the coaster is wooden or the rider asked for wooden. So if the rider types wooden (madeOfWood is true), ... || true is true for every coaster, so steel coasters like Steel Vengeance and Gatekeeper would wrongly print as "is wooden." The program still runs without crashing — it just mislabels coasters, which is a logic error.

why this answerA logic error runs fine but gives wrong output — so I swap one operator (&& → ||) that changes the meaning without breaking the build. I avoid something like changing < to <= in the loop, because that crashes with an out-of-bounds error — that's a runtime error, not a logic error.
2025 · 2CIf the list grows
the prompt, in shortSuppose another programmer adds several elements to the end of the list. Explain how the part (ii) code segment would need to change — or, if no change is needed, explain why.
answer

No change is needed. The loop is bounded by coasters.size(), not a fixed number like 4. .size() always reports the current number of elements, so adding coasters to the end just makes the loop run more times and print the new ones automatically.

One honest caveat: the parallel wooden list would need a matching 0 or 1 added for each new coaster so the indices still line up — but that's new data, not a change to the part (ii) code.

why this answerThis is the payoff of writing x < coasters.size() instead of x < 4. Because .size() adapts on its own, the answer is "no change," and the explanation is simply why: the loop bound tracks the list's real length.

2026 · 1Useful documentation
the prompt, in shortDescribe one piece of documentation you could include with your program, and explain how another programmer could use it to better understand a particular code segment.
answer

A header comment above findCoastersToRide describing what it does, its parameter, and its hidden dependency:

// findCoastersToRide(madeOfWood): walks the coasters list and prints each one,
// using the parallel "wooden" list to label it (1 = wooden, 0 = steel). Pass
// true when the rider asked for wooden, false for steel. Needs "wooden" to stay
// lined up with "coasters" -- same length, same order.
void findCoastersToRide(boolean madeOfWood) {

Reading just that comment, another programmer immediately learns three things they'd otherwise have to reverse-engineer: what madeOfWood means (true = the rider wants wooden), what the procedure produces (a printed line per coaster), and the rule that wooden must stay parallel to coasters. That lets them call and modify it safely without tracing the whole loop.

why this answerQ1 wants a specific documentation artifact tied to a specific segment — not "comments are good." A procedure header comment is the natural choice, and I connect it to this code's real confusion points: the meaning of the boolean and the parallel-list dependency.
2026 · 2AWhen the loop stops
the prompt, in shortFor the first iteration statement in your procedure, identify the variable(s) that determine when it stops, give specific values that stop it, and explain why. If there are no such variables, explain how it stops.
answer

The first iteration statement is the for loop:

for (int x = 0; x < coasters.size(); x++)

The values that control it are the counter x and coasters.size() (currently 4). The loop keeps going while x < coasters.size(). It stops when x reaches 4: at that point 4 < 4 is false, so the loop ends.

x starts at 0 and rises by 1 each pass, so it visits every index 0 through 3 exactly once before the test fails.

why this answerThe prompt asks for the controlling variable(s), an exact stopping value, and the reason. So I name both x and the .size() it's compared against, give x == 4 as the value where the condition flips false, and explain the counting.
2026 · 2BCan a call break it?
the prompt, in shortWrite a call to your procedure with specific argument(s) that the procedure accepts but that makes it behave incorrectly; describe and explain the incorrect behavior. If that's not possible, explain why.
answer

Not possible through the argument. madeOfWood is a boolean, so the only values the procedure accepts are true and false — and both produce the behavior the program intends:

findCoastersToRide(false) → prints only the steel coasters.
findCoastersToRide(true) → prints every coaster:

The coaster Steel Vengeance is steel. The coaster Beast is wooden. The coaster Gemini is wooden. The coaster Gatekeeper is steel.

Listing all coasters for a wooden request isn't a mistake — it's the intended design: a rider willing to take the rougher wooden coasters will also ride the smoother steel ones. Since neither boolean makes the procedure act against that design, no accepted argument produces incorrect behavior.

worth knowing

The picture flips if the intent changes. If the procedure were meant to show only the requested type, then findCoastersToRide(true) would behave incorrectly — it would list steel coasters too, because the second branch checks only wooden.get(x) == 0 and never madeOfWood == false. That stricter version, and the one-line check that enforces it, is Add-on 5 on page 4.

why this answerThe prompt allows "if that's not possible, explain why." With a boolean parameter and two intended outcomes, the honest answer is that nothing accepted breaks it — so the work is stating the procedure's intent precisely and showing both inputs satisfy it. Noting how a different intent would change the answer shows you understand why, not just that it passes.
2026 · 2CAbstraction & the list
the prompt, in shortExplain how the list uses abstraction to manage complexity. Then describe how you'd adjust the part (ii) code segment to keep the same behavior if the list were removed — or, if that isn't possible, explain why.
answer — the abstraction

One name, coasters, stands in for all four coaster names. coasters.get(x) and coasters.size() let a single loop handle any number of coasters, so the rest of the program never has to know how many there are or name them individually. The list hides that detail behind one name and an index — that's the abstraction.

answer — without the list

It is possible, but only by hard-coding a separate variable per coaster and giving up the loop. The part (ii) loop becomes a hand-written sequence — one if / else if block per coaster:

String coaster0 = "Steel Vengeance";  int wooden0 = 0;
String coaster1 = "Beast";            int wooden1 = 1;
String coaster2 = "Gemini";           int wooden2 = 1;
String coaster3 = "Gatekeeper";       int wooden3 = 0;

// part (ii) loses its loop -- one hand-written block per coaster instead:
if (wooden0 == 1 && madeOfWood == true) { println("The coaster " + coaster0 + " is wooden."); }
else if (wooden0 == 0) { println("The coaster " + coaster0 + " is steel."); }

if (wooden1 == 1 && madeOfWood == true) { println("The coaster " + coaster1 + " is wooden."); }
else if (wooden1 == 0) { println("The coaster " + coaster1 + " is steel."); }
// ...and so on for every coaster. Add one -> add another block by hand.

Same behavior, but every coaster you add now means more code written by hand. That repetition is exactly the complexity the list removes.

why this answer2C wants both halves: the abstraction explanation and a concrete without-list rewrite. So I explain "one name for many" plus get/size, then show the expanded version — individual variables and repeated if blocks with no loop — to make the saved complexity visible.
