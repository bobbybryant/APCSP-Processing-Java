The Number Oracle

The Oracle is thinking of a secret number between 1 and 1000. Your job is to build a game where a player tries to divine that number — with a limited number of guesses and hints along the way.

The pitch

You're building a complete, clickable guessing game from scratch. The player clicks a button, types a guess into a pop-up box, and the Oracle tells them whether they're too high, too low, or spot on. Guesses run out. Tension builds. Eventually the number is cracked — or the player loses.

This is the kind of project where every tool you've learned this semester shows up at once: variables, conditionals, loops, functions, and mouse input all working together in a single sketch.

What you'll practice

random() int / String / boolean if / else if / else IntList for loops functions mousePressed() hover detection

How this assignment works

The project is built in four levels. Each level adds features on top of the last, and each is worth points. Start at Level 1 and climb as far as you can — a working, basic game is worth far more than a fancy idea that doesn't run.

  • Level 1 — First Contact: a working guess-and-check loop 60 pts
  • Level 2 — Warmer, Colder: hints + limited guesses 20 pts
  • Level 3 — Keeping Track: a guess history + polish 20 pts
  • Level 4 — Make It Yours: your own extensions +15 bonus

Tip: get each level fully working and tested before you move up. Saving a copy of your sketch at the end of each level is a smart move — if Level 3 breaks something, you still have a clean Level 2 to turn in.

Level 01 · Foundation

First Contact

Required to pass · gets the core loop working
60 pts

Goal: a player can make one guess at a time and find out whether they got it right. No hints yet, no guess limit yet — just the bones of the game. By the end of this level your Oracle should pick a secret number, show instructions on screen, and respond to a clicked guess.

Requirements

  • The canvas opens at 600 × 400 and your instruction text is readable on screen. 5 pts
  • The Oracle picks a secret number using random(), and prints it to the console so you can test the game. 10 pts
  • The prompt is displayed on the canvas (something like "I'm thinking of a number between 1 - 1000. Can you guess what it is?"). 5 pts
  • A clickable button is drawn on the canvas. 10 pts
  • Clicking the button opens an input box, reads what the player typed, and turns it into a number. 15 pts
  • After a guess, the screen shows a correct message or an incorrect message. 15 pts
stuck? think about…

• A button is just a rect() you drew. "Clicking the button" means: in mousePressed(), check whether mouseX / mouseY landed inside that rectangle. You already did bounding-box checks like this.

• To pop up a text box, you'll use JOptionPane (see The Toolbox). Whatever the player types comes back as a String — you'll need to convert it to an int before you can compare it.

• Comparing two numbers for "are these equal?" uses ==, not =. (One equals sign assigns; two equals signs asks a question.)

test like a sneak Because you printed the secret number to the console, you can guess it on purpose to test your "correct!" message, and guess wrong on purpose to test your "incorrect" message. Don't leave that println() in your final game, though — it's a cheat code.
Level 02 · Core gameplay

Warmer, Colder

Turns a toy into an actual game
20 pts

Right now a player has to guess blindly — not much of a game. In this level you add the two things that make guessing games fun: helpful hints and a sense of pressure. The player should learn whether to go higher or lower, and they should be able to run out of guesses.

Requirements

  • On a wrong guess, the Oracle says whether it was too high or too low, and tells the player which way to go next. 8 pts
  • The game tracks how many guesses have been made and displays how many are left. 6 pts
  • When the player runs out of guesses, the game shows a "Game Over" message. 6 pts
stuck? think about…

• "Too high vs. too low" is a three-way decision: equal, greater than, or less than. That's an if / else if / else chain — exactly the shape you practiced in the conditionals unit.

• "Guesses left" = the max minus how many have been used. You'll need a counter variable that goes up by one each time a guess is entered.

• Want a message to appear on two lines? Drop \n in the middle of your String — for example "Too high.\nTry lower."

Decide what "out of guesses" means before you code it: does the player get exactly the max number of guesses, or one fewer? Test the edge.
Level 03 · Data & polish

Keeping Track

Full marks · the part that needs a list and a loop
20 pts

This is the level that separates a finished game from a great one. The Oracle should remember every wrong guess and show them on screen, notice when a player repeats a guess, and give that satisfying little visual cue when the mouse is over the button. To pull this off you'll need a place to store many numbers — that's where IntList comes in.

Requirements

  • Every incorrect guess is stored and the full list is displayed on screen, one under the other. 8 pts
  • If the player guesses a number they've already tried, the Oracle tells them — and that repeat does not cost them a guess. 8 pts
  • The button changes color when the mouse hovers over it. 4 pts
stuck? think about…

• One int variable holds one number. To hold a growing collection of wrong guesses, you need an IntList — see The Toolbox for how to make one, add to it, and read from it.

• To display the whole list, walk through it with a for loop. Your loop counter starts at 0 and goes up by 1 until you reach the size of the list. Use the counter to figure out the y-position of each line so they stack instead of landing on top of each other.

• To check "have they guessed this before?", loop through the stored guesses and compare each one to the new guess. A boolean flag (start it false, flip it to true if you find a match) is a clean way to remember the answer.

• The hover effect is the same bounding-box idea as the button click — but checked every frame in draw(). If the mouse is inside the rectangle, fill it one color; otherwise, fill it another.

the tricky one The "don't punish repeats" rule is the sneakiest requirement in the project. Go back and play the target game again, repeat a guess on purpose, and watch the guess counter. Match that behavior exactly. Both halves count: it should say something AND it should not lose a guess.
Level 04 · Challenge

Make It Yours

Bonus · only attempt after Levels 1–3 work
+15 bonus

You've built the assigned game. Now make it your game. Pick from the menu below or invent your own feature — each meaningful, working addition is worth up to +5 bonus points, to a maximum of +15. The only rule: it has to actually run, and it has to clearly do something. Half-finished extensions earn nothing.

Warmer / colder

Change the message (or the background color) based on how close the guess is — "burning hot" when they're within 10, "freezing" when they're way off.

Play again

Add a reset button that picks a fresh secret number, clears the guess list, and restarts the game without re-running the sketch.

Difficulty modes

Let the player pick easy / medium / hard, which changes the number range, the guess limit, or both.

Guard the input

Handle bad input gracefully — what should happen if the player types letters, leaves it blank, or guesses outside the range?

Win celebration

Add a visual payoff on a correct guess — confetti dots, a color flash, a "you did it in N guesses!" stat.

Your idea

Got something else in mind? Go for it. Leave a comment at the top of your code describing what you added so it gets graded.

Reference

The Toolbox

Three tools in this project might be new or rusty. Here's just enough syntax to use each one — how you wire them together to make the game work is the part you figure out.

new tools for this build

1 · Asking the player to type something

Processing can pop up a little input box using JOptionPane. Whatever the player types comes back as text (a String), so you convert it to a number before you can compare it.

import javax.swing.JOptionPane;   // goes at the very TOP of your sketch

String input = JOptionPane.showInputDialog("Enter your guess:");
int guess = int(input);   // turn the typed text into a whole number

Once it's an int, comparing it to your secret with == works fine. (Comparing two Strings would be a different story — that needs .equals() — but here you're comparing numbers.)

2 · Storing a whole pile of numbers

One int holds one number. An IntList holds as many as you want and grows as you add to it — perfect for a list of wrong guesses.

IntList guesses = new IntList();   // an empty list of ints
You writeWhat it does
guesses.append(42)adds the number 42 to the end of the list
guesses.size()gives back how many numbers are in the list right now
guesses.get(0)gives back the number at position 0 (the first one — counting starts at 0)

3 · Picking the secret number

int secret = int(random(1, 1000));

random(1, 1000) hands you a decimal that's at least 1 and below 1000; wrapping it in int(...) chops off the decimal. Worth a moment of thought: with that upper number not included, which values can actually come out?

Walking through a list (the loop shape you'll reuse)

Whenever you need to touch every item in a list — like to draw all the wrong guesses, or to search them for a repeat — you use a counting loop. The shape is always the same:

for (int i = 0; i < howMany; i = i + 1) {
  // use i in here -- this runs once for each item
}

It starts at 0, climbs by 1, and stops before it runs off the end. Any math you need — like figuring out the y-position of each line you draw — happens inside the loop, using i.

Code craft — this gets looked at too

A working game earns the points in each level. How you write it affects your score too, in small but real ways. Aim for these:

  • Build with helper functions. Give each job its own function — one for entering a guess, one for checking duplicates, and so on. Don't cram everything into draw().
  • Don't hand a function arguments it never uses. If a function doesn't need a piece of information to do its job, don't pass it in.
  • Name things for what they hold. secretNumber and guessesLeft, not x and y2.
  • Loops start at 0 and count up by 1, with the positioning math inside the loop body.

How It's Graded

Points come from features that actually work, level by level. The levels stack, so finishing more of them means a higher score:

If you finish… Your score

Each level breaks down into the specific point values listed in its section above. Inside a level, partial credit is real — a feature that mostly works earns most of its points.

where points slip away

• A feature that crashes or doesn't run can't earn its points — test everything before you submit.

• Output bugs count: guess counts that go wrong, wrong guesses that don't show up, "too high" when it should say "too low."

• Craft (see the Toolbox): everything jammed into draw(), functions handed values they never use, or single-letter mystery variable names will cost you a little.
