01 · The Big Picture

One idea ties the whole project together

Almost everything you'll do here is a version of the same move, repeated:

Set a few variables  →  hand them to helper functions as arguments  →  each helper uses if statements to decide what to draw.

Your setup() (or draw()) is basically a control panel. It holds the dials for your world — the season, the hour, how many clouds — and then calls helpers to build the scene. It should contain only variable assignments and function calls. No fill(), no rect() living directly in there.

void setup() {
  size(800, 600);

  int timeOfDay = 14;          // the dial: 0 to 23
  String season = "SUMMER";    // the dial: SUMMER / SPRING / WINTER / FALL
  int numClouds = 3;           // the dial: how many of something

  drawSky(season, timeOfDay);  // hand the dials to the helpers
  drawLandscape(season, timeOfDay);
  drawClouds(numClouds);
}

notice: the panel just sets values and calls helpers — nothing is drawn here directly

The trickier part is the arrow in the middle: which helper actually needs which dial? A function should only receive the arguments it truly uses to do its one job. Try it — tap a variable and see who needs it:

Tap a variable
drawSky(...)

drawLandscape(...)
Ground color and seasonal ground details.

Bare, leafy, or snowy depending on the season.
Each function should take in only what it needs — that's worth points on the rubric.
Why this matters

Passing a function exactly the arguments it needs (and no extras) is literally a graded line on your rubric. A drawClouds() that demands the season is a hint that the job isn't cleanly separated yet.

02 · Plan Before You Code

Sketch first. The code is the easy part later.

The students who struggle most are usually the ones who open Processing and start typing. The ones who finish calmly drew a plan first. Spend ten minutes here before any code:

✏️

Sketch the scenes

Doodle what SUMMER noon looks like vs WINTER night. You don't need 96 drawings — a few rough sketches reveal what changes and what stays.

🧱

List the layers

Sky, ground, trees, an object or two, then small surprises. That list becomes your function calls.

🔧

Name the helpers

Write the function names you'll need before writing them: drawSky, drawTree, drawSnowman…

Order is everything: last drawn sits on top

Processing paints in the order you call things, like stacking sheets of paper. The first call lands on the bottom; the last call sits on top. So your sky and ground come first, your tiny details come last — otherwise the ground paints right over your snowman. Step through it:

Your draw() calls, in order
  • 1.drawSky(...)
  • 2.drawLandscape(...)
  • 3.drawTrees(...)
  • 4.drawSnowman(...)
  • 5.drawStars(...)

03 · Your Variables

Three or more dials, set in the right place

The assignment asks for at least three variables. These three are required:

🕐

timeOfDay

An int from 0 to 23. Drives sunrise / day / sunset / night.

🍂

season

A String: "SUMMER", "SPRING", "WINTER", or "FALL".

🔢

num___

An int for how many of something — clouds, stars, leaves. Plus anything else you need.

The #1 thing graders look for

Declare your variables locally, inside setup() (or draw()) — not globally at the top of the file. On the rubric, local declarations earn full marks; global ones drop you a level even if everything else is perfect. And every variable you declare must actually get used.

Naming rules (same as always)

Start with a letter, no spaces or symbols except the underscore, and pick a name that says what it holds. numClouds tells a reader something; x2 doesn't. Clear names are graded too.

The one operator mistake that bites everyone

When you compare things inside an if, use == (double). A single = means "put this value in here." Mixing them up is the most common bug in this whole project. Decide what each line is doing:

Comparing (==) or assigning (=) ?
if (season == "WINTER")
numClouds = 5;
if (timeOfDay = 18)
String season = "SPRING";
Tap a choice for each line. Single = sets a value; double == checks equality.
04 · Thinking in Ranges

You don't need 24 looks — you need a few ranges

The clock runs 0 to 23, but nobody wants to write 24 different skies. Group the hours into a handful of moods — night, sunrise, midday, sunset, night — and let an if / else if / else chain sort each hour into exactly one of them.

The chain reads top to bottom and stops at the first match. That's why timeOfDay < 6 quietly means "0 through 5": anything below 6 already got caught. Drag the hour and watch which branch wins:

DAYTIME
14:00
if (timeOfDay < 6) { // night } else if (timeOfDay < 10) { // sunrise } else if (timeOfDay < 18) { // daytime } else if (timeOfDay < 20) { // sunset } else { // night again
Graded: efficient conditionals

The rubric rewards the "most efficient use of if/else if/else." A tidy chain like the one above beats five separate if statements, because the chain guarantees exactly one branch runs. Keep your braces lined up and your conditions in a sensible order (smallest range first).

Do the same trick for the seasons inside each helper: one if/else if chain comparing season to each of the four strings.

05 · Helper Functions

Lots of small functions, each with one job

A good helper function does one focused thing, has a name that says what it does, and uses the arguments it's given. drawTree() draws a tree. drawSnowman(size) draws a snowman of that size. You're allowed — encouraged — to call other helpers, the way drawLandscape() might call drawTree(), which calls drawOneLeaf().

A strong helper

Clear name · does a single task · takes only the arguments it needs · actually uses those arguments inside its body.

A weak helper

Vague name · tries to do five things · takes an argument it never looks at · or holds code that should be its own separate helper.

Two graded traps, side by side

Function Calling: if drawing commands like fill() or rect() sit loose in draw() instead of inside a helper, you drop from full marks. Helper Functions: if a function takes season but never checks it, the rubric calls that "not properly using arguments."

Sorting test: where does each line belong?

Your control panel holds variables and function calls. Everything that actually paints pixels lives inside a helper. Sort each line into the right home:

String season = "WINTER";
drawSky(season, timeOfDay);
fill(255, 200, 0);
triangle(655, 420, 740, 415, 695, 360);
int numStars = 30;

06 · Easter Eggs

The surprises that only show up sometimes

This is the playful, heavily-graded part. An easter egg is something that appears only in a certain season or time — snowflakes in winter, a moon and stars at night, a squirrel in spring. The magic word is only: a bird that's always on screen isn't an easter egg, it's just a bird.

Mechanically, an easter egg is just a helper call wrapped in a condition:

if (season == "WINTER") {
  drawSnowflakes();   // only ever happens in winter
}
How easter eggs are scored

Top marks go to ideas that are thoughtful, clever, and well-executed, and that appear only when appropriate. "Too few or too simple" lands in the middle; missing them is zero. Aim for a couple you're genuinely a little proud of.

08 · Before You Submit

The pre-flight check

Most lost points aren't about hard logic — they're the small stuff that's easy to forget at the end. Tick these off (tap to check) right before you turn it in:

Last look before takeoff

  • Run auto-format: CMD + T right before submitting. It's a free point.
  • All 3+ variables declared locally in setup() (or draw()) — and every one is actually used.
  • setup() / draw() holds only variable assignments and function calls. No stray fill() or rect() in there.
  • Each helper does one job, has a clear name, and uses the arguments you pass it.
  • Comparisons use ==, never =; the hours are grouped into ranges with if/else if/else.
  • At least a couple of easter eggs that show up only in the right season or time.
  • Four screenshots of different scenes, dropped into one Word/Google Doc and submitted with your code.
