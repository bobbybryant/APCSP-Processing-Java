Functions & the Anatomy of a Program

A function is a named block of code that does one job. Once you learn them, every line of code you write will live inside one.

no more loose code — it all goes in a function now! ✓

01 · THE BIG SHIFT

Everything now lives inside a function

Up to now you could write commands one after another and they'd just run. From here on, Processing needs your code organized into functions. The program always begins in a special function called void setup().

✗ Old way (loose code)

size(400, 300);
background(135, 206, 235);
ellipse(200, 150, 80, 80);
// floating commands — not allowed anymore

✓ New way (inside setup)

void setup() {
  size(400, 300);
  background(135, 206, 235);
  ellipse(200, 150, 80, 80);
}
Where it all starts: setup(). void setup() is the special function Processing runs automatically when your sketch begins. It runs once, top to bottom — perfect for a drawing that's set up a single time. Every other function you write gets called from inside setup() (or from another function that setup() called).
02 · ANATOMY

The four parts of a function

Every function you define has the same skeleton. Hover each piece to see what it does:

void drawCloud(float x, float y) { ... }
void Return type — what the function gives back. void = nothing.
drawCloud Name — how you call the function later.
(float x, float y) Parameters — the inputs the function accepts.
{ ... } Body — the actual job, run top to bottom.

return type · name · parameters · body

The return type can be any of the 9 data types from the variables unit — int, float, double, boolean, char, String, byte, short, long — plus void, for 10 total options. The type you choose just has to match what the function actually hands back: a whole number → int, true or false → boolean, text → String, and so on. If it hands back nothing at all, that's void.

How to decide what kind of function you need

Ask yourself two questions. Every possible answer leads to one of the four combinations covered in sections 03–05.

03 · NO PARAMETERS, NO RETURN

The simplest function: just do a job

Some functions take no inputs and give nothing back — they just do something, like draw a fixed shape. Because it has no parameters, drawSun() always draws the exact same sun in the exact same spot every time you call it.

void drawSun() {
  fill(255, 214, 10);   // yellow
  noStroke();           // no border
  ellipse(230, 60, 80, 80);
}

void setup() {
  size(300, 220);
  background(135, 206, 235);
  drawSun();   // call it by name
}

To call a function, write its name followed by ();. No parameters means nothing goes in the parentheses.

drawSun() — same result every time
04 · WITH PARAMETERS

Parameters let one function do many things

Our class example: drawCloud(x, y) draws 7 white circles with no border to make a single fluffy cloud — placed wherever you ask. The 7 circles are positioned relative to x and y, so changing the inputs slides the whole cloud.

void drawCloud(float x, float y) {
  fill(255);          // white
  noStroke();         // no border on any circle
  circle(x,      y,      44);   // center puff
  circle(x - 30, y + 6,  36);   // left
  circle(x + 30, y + 6,  36);   // right
  circle(x - 15, y - 12, 40);   // top-left
  circle(x + 15, y - 12, 40);   // top-right
  circle(x - 48, y + 9,  28);   // far left
  circle(x + 48, y + 9,  28);   // far right
}

circle(x, y, d) is the shortcut for a perfect circle: center (x, y) and a single diameter d — no need to repeat width and height like ellipse.

Defining vs. calling. You define a function once (with its types and parameters). You can call it as many times as you like — and each call can pass different arguments: drawCloud(90, 80); then drawCloud(250, 60);.
05 · RETURN TYPES

When a function hands a value back

void functions do something (like draw). But sometimes you want a function to compute and hand back an answer. That's what a return type is for. Instead of void, you write the type of value it returns, and you use the return keyword inside.

int addScores(int a, int b) {
  return a + b;   // hands back the total
}

void setup() {
  int total = addScores(7, 5);
  println("Total is now: " + total);   // → Total is now: 12
}

The returned value can be stored in a variable, printed, or used right away. Here we feed the total straight into a bar's width — and print it so we can see it.

Printing to the console: print and println

println(...) writes text to the console — the message area at the bottom of the Processing editor, separate from your drawing. It's how your program talks back to you. print(...) does the same thing but stays on the same line; println adds a line break after it, so each message lands on its own line. You'll reach for println most of the time.

print("Score: ");
println(42);          // Score: 42   (print stays put, then println ends the line)
println("Hello!");    // Hello!      (its own line)

The real reason we lean on it: debugging. When a result looks wrong, printing a value shows what your program is actually working with instead of you guessing. A great use is checking what a function handed back:

int total = addScores(7, 5);
println(total);             // 12  — confirms what came back

// or test a function call directly, without even storing it:
println(addScores(10, 4));  // 14

You can also drop a println inside a function to watch the values flowing through it while you test:

int addScores(int a, int b) {
  println("addScores got a=" + a + ", b=" + b);  // peek at the inputs
  return a + b;
}

Now the console prints something like addScores got a=7, b=5 every time the function runs — instant proof of the inputs it received. Once everything works, you delete the debug line.

void vs. a return type. Use void when the function's job is to act (draw, print, move). Give it a real type (int, float, boolean, String…) when its job is to calculate and return an answer. A non-void function must use the return keyword.
06 · ORDER OF EXECUTION ★

Where functions live vs. when they run

This trips everyone up: a function's position in your file does not decide when it runs. You can define functions above, below, or anywhere around setup(). Processing reads them all, but nothing happens until a function is called by name. Then execution jumps in, runs that body top to bottom, returns to where it left off, and continues.

Notice below: drawSky() is defined above setup, and drawSun() / drawCloud() are defined below it — yet the screen fills in the order they're called. Press Step (or Play) and watch.

The mental model. Definitions can be anywhere. setup() runs top to bottom. Each call jumps into the named function, runs it fully, then comes back to the next line. Calling drawCloud(90, 80) and then drawCloud(250, 60) runs the same body twice with different inputs.
