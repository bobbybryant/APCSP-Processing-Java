Making decisions with conditionals

Now that a sketch knows where the mouse is, it can start asking questions — "is the mouse on the left?" — and act on the answer. That's what if-statements do. Every panel below is live, so poke at it.

01 — THE QUESTIONS

Relational operators ask yes/no questions

Every conditional starts with a question that is either true or false — a Boolean expression. Relational operators are how we compare two values to build that question.

  • >  greater than  ·  <  less than
  • >=  greater than or equal  ·  <=  less than or equal
  • ==  equal to  ·  !=  not equal to

Watch out — == is not = The single biggest beginner bug. = assigns a value; == compares two values. In a condition you almost always want ==.
02 — RUN, OR SKIP

The simple if

An if guards a block of code. If the condition is true, the block runs. If it's false, Processing skips the whole block and moves on — no "otherwise," nothing else happens.

Key idea A lone if has exactly one outcome to offer: do the block, or don't. "False" just means "skip it" — there's no backup plan unless you add one.
03 — ONE OR THE OTHER

if / else: always pick a path

Add else to give the "otherwise" path. Now exactly one of the two blocks always runs: the if block when the condition is true, the else block every other time.

Key idea if alone = "maybe do something." if / else = "definitely do one of these two." The else never has its own condition — it's the catch-all.
04 — MANY PATHS

Chaining with if / else if / else

When there are more than two cases, chain them. Processing checks each condition in order from the top, runs the first one that's true, then skips all the rest. The final else catches anything left over.

05 — ORDER MATTERS

Checking conditions in the right order

In a chain, Processing tests conditions top to bottom and runs the first one that's true. So the order you write them in can change the answer — get it wrong and some branches become impossible to reach.

Classic example: greeting someone based on the hour, on a 24-hour clock (0–23).

Key idea When chain conditions overlap, order them from most specific to least specific. Each else if only ever sees the cases the branches above it didn't already claim.
06 — COMBINING CONDITIONS

and · or · not

Logical operators glue yes/no questions together (or flip them), so one condition can test several things at once.

  • &&  and — true only when both sides are true.
  • ||  or — true when at least one side is true.
  • !  not — flips a Boolean: !true is false, !false is true.

07 — TWO WAYS TO COMBINE

Nested vs compound conditionals

Sometimes several things need to be true at once. There are two ways to write that, and it's worth knowing both.

  • Compound — one if with conditions joined by && or ||.
  • Nested — an if placed inside another if.

The classic case: is the mouse hovering a rectangle?

A rectangle covers a range of X and a range of Y, so the mouse is "inside" only when it's past all four edges. Using mouseX and mouseY, here it is both ways:

Which should I use? For a plain "all of these must be true," compound with && is cleaner and easier to read. Reach for nesting when you also want to act at the in-between step — e.g., "if the mouse is in the right column, dim it; and if it's also in the right row, select it." Nesting lets each layer do its own thing.
Good to know A nested if with nothing else around it does the exact same job as a single compound &&. They're logically equivalent — so when you only need the innermost result, prefer the compound version to keep the code flat.
08 — PUTTING IT TOGETHER

Booleans inside functions

A Boolean can be stored in a variable to remember an answer, and a function can return one. That lets you give a complicated condition a clear name — and then your if-statements read like plain English.

Here's a real button: isOverButton() bundles a relational && chain and hands back true/false. We use it to highlight on hover, and a lampOn Boolean (flipped with !) decides whether the lamp's glow gets drawn at all.

★ — QUICK RECAP

Everything, on one screen

  • A condition is a Boolean expression — it's either true or false.
  • Relational ops compare values: > < >= <= == !=. Remember == compares, = assigns.
  • if runs a block when true, skips it when false. No backup plan on its own.
  • if / else always runs exactly one of two blocks.
  • if / else if / else picks one of many — the first true branch wins, the rest are skipped.
  • Order matters: in a chain, check the most specific condition first — a broad condition placed early can swallow cases meant for a later branch (dead code).
  • Logic: && needs both true, || needs one, ! flips a Boolean.
  • Compound vs nested: join tests in one if with &&/|| (compound), or put an if inside another (nested). Hovering a rectangle needs all four edges true.
  • Booleans can be stored in variables and returned by functions to decide whether something happens.
